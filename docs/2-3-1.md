## 1. 이 문서를 한눈에

### 이 문서가 정의하는 것

이 문서는 **"러닝 영상이 분석 대상으로 적합한지 판단하는 입구 단계"** 의 정책을 정의합니다. AI 파트 안에서 이 문서는 다음 세 가지의 **단일 정답 위치(single source-of-truth)** 역할을 합니다.

- **영상 메타데이터 임계값** (영상 길이, 초당 프레임 수, 해상도)
- **MVP 분석 범위 정의** (어떤 영상까지 받을 것인가)
- **사용자 촬영 가이드 메시지** (촬영 시작 전 사용자에게 보여줄 안내)

### 이 문서가 정의하지 않는 것

다음 항목은 다른 문서에 정의되어 있으니 그쪽을 참조해주세요.

| 알고 싶은 것 | 가야 할 문서 |
| --- | --- |
| 영상 화질이 낮을 때(블러, 어두움)는 어떻게 처리하나요? | **2-3-2** (영상 전처리 / Frame-level 품질 검사) |
| Foot Strike Pattern이나 무릎 굴곡각은 어떻게 계산되나요? | **2-3-4** (핵심 자세 지표 산출) |
| MediaPipe로 추출한 관절 좌표의 신뢰도(visibility)는 어떻게 검사하나요? | **2-3-5** (측정 품질 판단) |
| 분석 결과 상태(success / low_confidence / failed)는 무엇 기준으로 결정되나요? | **2-3-5** (8장 reason code) |
| `too_short`, `low_fps` 같은 reason code의 정확한 의미는? | **2-3-5** (8장 reason code) |
| 분석 결과 JSON은 어떤 형식으로 응답되나요? | **2-3-7** (자세 분석 결과 구조화) |
| WebSocket 연결 설정, frame 전송 방식, 에러 코드는? | 백엔드 WebSocket 설계 문서 |

### 다른 파트가 알아두면 좋은 핵심 결정 3가지

이 문서에서 결정된 사항 중, 다른 파트(백엔드, 프론트엔드)에 영향을 주는 핵심 정책은 다음과 같습니다.

1. **백엔드는 WebSocket 기반 실시간 스트림 구조를 채택한다.** Android에서 카메라 frame을 binary로 Spring에 전송하면, Spring은 이를 AI 서버에 relay하고, AI 서버는 frame별 추론 결과 JSON을 Spring을 거쳐 Android로 전달한다. 영상 파일 업로드 모드는 사용하지 않는다.
2. **5초 미만의 누적 분석 시간은 분석 거부.** 5초 이상 10초 미만은 신뢰도 낮음 표기, 10초 이상이 권장 범위입니다. (프론트엔드: 촬영 가이드에 "10초 이상 달려주세요" 표기)
3. **다중 인물 즉시 거부 정책은 사용하지 않는다.** "한 명만 들어오게" 가 아니라 "분석 대상자가 화면 중앙에 가장 크게 보이도록" 가이드로 안내합니다.

## 2. MVP 분석 범위

본 시스템은 다음 범위의 영상만 분석 대상으로 한다.

### 2-1. 입력 환경

| 항목 | MVP 범위 | 비고 |
| --- | --- | --- |
| 촬영 환경 | 러닝머신 (실내) | 실내 통제 환경에서 분석 안정성 확보. 야외 러닝은 향후 확장 예정. |
| 촬영 시점 | 측면 고정 촬영 (`view_type = side`) | 정면/후면 촬영은 향후 확장 예정. |
| 입력 방식 | 실시간 스트림 (WebSocket binary frame 전송) | Android 카메라 frame을 단일 frame 단위 binary로 전송. 영상 파일 업로드 방식은 사용하지 않는다. |

### 2-2. 입력 처리 구조

본 시스템의 입력 처리는 **Android → Spring → AI 서버 3-tier WebSocket 릴레이 구조**로 동작한다.

```
[Android 카메라]
      │
      │  binary frame (JPEG 압축)
      │  WebSocket 연결: /ws/chobo-runner
      ▼
[Spring 백엔드]
      │
      │  binary frame relay (VideoRelayHandler)
      │  Android 세션 ↔ AI 세션 매핑
      ▼
[AI 서버 (Python WebSocket)]
      │
      │  frame별 추론 결과 JSON 응답
      ▼
[Spring 백엔드]
      │
      │  JSON relay
      ▼
[Android]
```

각 tier의 책임은 다음과 같다.

| Tier | 책임 | 본 문서와의 관계 |
| --- | --- | --- |
| Android | 카메라 frame 캡처, JPEG 압축, WebSocket binary 전송 | 본 문서 4장 촬영 가이드를 UI에 반영 |
| Spring | Android ↔ AI 서버 세션 릴레이, 메시지 크기 제한 관리 | 본 문서 메타데이터 임계값을 직접 검증하지 않음 (AI 서버에서 검증) |
| AI 서버 | frame 디코딩, 메타데이터 검증(본 문서 3장), Pose 추출, 지표 산출 | 본 문서가 정의하는 모든 검증의 실제 수행 위치 |

### 2-3. 분석 대상자 정책

본 시스템은 한 명의 사용자(분석 대상자) 분석을 전제로 한다. 다만 헬스장 러닝머신 환경의 특성상 주변 인물이 일부 화면에 포함될 수 있으므로, 다음 정책을 적용한다.

- **MVP 정책**: 분석 대상자가 화면 중앙에 가장 크게 위치하도록 사용자에게 가이드를 제공한다.
- **간섭 처리**: 주변 인물로 인해 분석 대상자의 추적이 불안정해지는 경우의 처리 방식은 2-3-5 문서의 분석 대상자 추적 안정성 정책을 따른다.

> **결정 사항**: 다중 인물이 화면에 보인다는 사실 자체로 분석을 즉시 거부하지 않는다. 헬스장 환경의 사용성을 고려한 결정이며, 실제 거부 여부는 추적 안정성 기반으로 2-3-5에서 판단한다.
> 

### 2-4. 부분 결과 미제공 정책

1차 MVP에서는 핵심 지표 3개(Foot Strike Pattern, Initial Knee Flexion, Trunk Lean)가 **모두 계산 가능한 경우에만** 최종 분석 결과를 사용자에게 제공한다.

| 상황 | 처리 |
| --- | --- |
| 핵심 지표 3개 모두 계산 가능 | 분석 결과 제공 (`success`) |
| 일부 지표 계산은 가능하나 신뢰도 부족 | 재촬영 안내 (`low_confidence`) |
| 핵심 지표 중 하나 이상 계산 불가 | 분석 불가 안내 (`failed`) |

부분 결과 제공(예: Trunk Lean만 보여주기)은 1차 MVP에서 제외하며, 향후 확장 시 검토한다. 상태값별 정량 기준 및 분기 룰은 2-3-5 문서에 정의되어 있다.

> **실시간 스트림 모드 보충**: 실시간 스트림 구조에서는 frame이 도착할 때마다 추론 결과가 응답되지만, 누적 stride 수가 충분해질 때까지는 "분석 중" 상태가 유지된다. 본 문서의 부분 결과 미제공 정책은 "최종 누적 분석 결과 응답 시점"에 적용된다.
> 

### 2-5. 분석측 단일값 정책

본 시스템은 측면 단일 카메라 환경에서 **분석측 한쪽 다리만 측정**한다. 좌우 분리 측정은 하지 않으며, 분석측 결정은 영상 시작 후 1~2초 동안의 visibility 평균이 더 높은 측으로 자동 결정한다.

분석측 결정 알고리즘은 2-3-4 문서의 분석 대상 측 선택 정책을 따른다. 응답 메시지의 형식은 2-3-7 문서에 정의되며, 분석측을 명시하는 `analysis_side` 필드를 포함한다.

좌우 비대칭 분석은 후면 카메라 도입 또는 양측 측면 카메라 환경에서 의미 있는 분석이 가능하므로, 향후 확장 예정이다.

## 3. 입력 영상 메타데이터 기준

본 장은 분석 대상 영상의 메타데이터(누적 분석 시간, Effective FPS, 해상도, 누적 frame 수)를 검증하는 기준을 정의한다. 본 장의 모든 임계값은 **초기 default 값**이며, 파일럿 데이터 검증을 통해 보정한다.

### 3-1. 메타데이터 임계값 표

| 검증 항목 | `failed` 기준 | `low_confidence` 기준 | 권장 기준 | 미충족 시 reason code |
| --- | --- | --- | --- | --- |
| **누적 분석 시간** | 5초 미만 | 5초 이상 ~ 10초 미만 | 10초 이상 | `too_short` |
| **Effective FPS** | 24fps 미만 | 24fps 이상 ~ 30fps 미만 | 30fps 이상 | `low_fps` |
| **해상도 (긴 변 기준)** | 720p 미만 | (별도 임계 없음) | 720p 이상 | `low_resolution` |
| **누적 frame 수** | 120 미만 | 120 이상 ~ 240 미만 | 240 이상 | `too_short` |

> **Effective FPS 정의**: Android 카메라의 nominal fps(촬영 설정값, 예: 30fps)가 아니라, **AI 서버가 분석 종료 시점까지 실제로 수신·디코딩에 성공한 frame 수를 누적 분석 시간으로 나눈 값**이다.
> 
> 
> ```
> effective_fps = received_frames / analysis_duration_sec
> ```
> 
> 실시간 스트림에서는 JPEG 인코딩 + WebSocket 전송 + Spring 릴레이 + AI 서버 디코딩 과정에서 frame drop이 발생하므로, nominal 30fps로 캡처해도 effective fps는 18~22fps 수준으로 떨어질 수 있다. 본 문서의 모든 FPS 임계값은 Effective FPS 기준이다.
> 

### 3-2. 메타데이터 검사 결과 충돌 시 우선순위

여러 메타데이터 검사 결과가 서로 다른 상태값을 가리킬 때, **가장 나쁜 상태를 최종 상태로 채택**한다.

```
우선순위:  failed  >  low_confidence  >  success
```

**예시**: 누적 분석 시간이 6초(`low_confidence`)이고 누적 frame 수가 90개(`failed`)인 경우, 최종 상태는 `failed`로 결정된다.

이 우선순위 규칙은 메타데이터 검증뿐 아니라, 메타데이터 검증 결과와 후속 단계(2-3-2 / 2-3-5)의 검증 결과 사이에서도 동일하게 적용된다. 즉, 어떤 단계에서든 한 번이라도 `failed`가 발생하면 최종 상태는 `failed`이다.

### 3-3. 임계값 근거

- **누적 분석 시간 5초 미만 = `failed`**: 평균 stride time 0.7초 기준으로 IC 후보가 약 7개 미만 검출됨. 2-3-4 문서 8장에서 정의한 "최소 유효 stride 3개" 조건을 만족하기 어려움.
- **누적 분석 시간 5~10초 = `low_confidence`**: IC 후보 자체는 확보 가능하나, 시작/끝 구간을 분석에서 제외하면 중간 구간 후보가 부족할 가능성이 있음.
- **누적 분석 시간 10초 이상 권장**: 충분한 stride 누적이 가능하며, 일시적 outlier가 발생해도 중앙값으로 흡수 가능.
- **Effective FPS 24 미만 = `failed`**: IC 검출 시점 분해능이 1프레임당 42ms 이상이 되어, IC 시점 보정의 의미가 사라짐 (2-3-4 문서 3장 참조).
- **해상도 720p 권장**: MediaPipe Pose가 내부적으로 입력을 변환하므로 1080p 대비 정확도 차이가 크지 않으며, 모바일 업로드 및 WebSocket 전송 비용 측면에서 720p가 균형점.
- **해상도 기준은 필요조건**: 본 해상도 기준은 영상이 분석 대상이 되기 위한 **필요조건**이다. "분석 대상자가 화면에서 충분히 크게 보이는가" (사람 bbox 비율, 발 landmark 가시성, 전신 포함 비율)의 충분조건 검사는 2-3-5의 visibility / bbox 비율 검사에서 수행한다. 즉, 720p를 만족해도 사람이 너무 멀리 찍혀 있으면 2-3-5에서 별도로 거부될 수 있다.

### 3-4. 검증 시점 분류

본 문서의 메타데이터 검증은 **검증 시점에 따라 즉시 검증 / 누적 검증** 두 가지로 분리된다. 후속 검증(블러, visibility, 측면 구도 등)은 별도 문서에서 다룬다.

### 즉시 검증 — 첫 frame 도착 시점

| 검증 항목 | 판정 가능 시점 | 미충족 시 |
| --- | --- | --- |
| 해상도 (긴 변 720p 미만) | 첫 frame decode 직후 | `failed` 즉시 응답, 이후 단계 진입 안 함 |
| Frame decode 가능 여부 | 첫 frame 수신 직후 | `failed` 즉시 응답, 이후 단계 진입 안 함 |

이 두 항목은 단일 frame만으로 판정 가능하므로, AI 서버가 첫 frame을 수신한 직후 즉시 검증한다. 미충족 시 후속 단계(Pose 추출, 지표 산출)로 진입하지 않는다.

### 누적 검증 — 분석 종료 시점

| 검증 항목 | 판정 시점 | 처리 |
| --- | --- | --- |
| 누적 분석 시간 | 분석 종료 시점 | 우선순위 규칙으로 최종 상태 결정 |
| Effective FPS | 분석 종료 시점 | 우선순위 규칙으로 최종 상태 결정 |
| 누적 frame 수 | 분석 종료 시점 | 우선순위 규칙으로 최종 상태 결정 |

이 세 항목은 분석 종료 시점에 누적 데이터로 판정한다. 분석 도중에는 "분석 중" 상태가 유지되며, 종료 시점에 3-2의 우선순위 규칙을 적용해 최종 상태를 결정한다.

### 후속 검증 — 본 문서 영역 외

다음 검증은 본 문서가 직접 정의하지 않으며, 해당 문서를 참조한다.

| 검증 영역 | 정의 위치 |
| --- | --- |
| Frame-level 화질 검사 (블러, 휘도, 흔들림) | 2-3-2 |
| Landmark visibility, 측면 구도, 분석 대상자 추적 안정성 | 2-3-5 |
| Stride 단위 안정성, IC 후보 신뢰도 | 2-3-4 (산출 기준), 2-3-5 (사용 정책) |

### 3-5. 분석 종료 시점 정의

누적 검증을 위해서는 "분석 종료 시점"이 명확히 정의되어야 한다. AI 서버는 다음 두 가지 trigger 중 하나가 발생하면 분석 종료로 판정한다.

| Trigger | 판정 기준 |
| --- | --- |
| **사용자 명시적 종료** | Android에서 분석 종료 메시지를 전송하고, Spring을 거쳐 AI 서버에 도달한 시점 |
| **자동 종료 (timeout)** | AI 서버가 마지막 frame 수신 후 **2.0초** (default) 동안 추가 frame이 도착하지 않으면 분석 종료로 판정 |

> **2.0초 기준의 의미와 보정 계획**:
> 
> - 일반적인 LTE/Wi-Fi 환경에서 frame 간격은 30~60ms 수준이므로, 2.0초는 그 30배 이상에 해당한다. 일시적 네트워크 지연으로 오인할 가능성이 낮은 충분한 buffer이다.
> - 이 timeout 값은 백엔드 WebSocket 설계 문서의 heartbeat 정책 및 재연결 정책과 동기화한다. 백엔드 측 heartbeat interval보다 짧으면 실제로는 연결이 살아있는데 분석 종료로 오판정할 수 있다.
> - 파일럿 테스트 결과에 따라 1.5초 ~ 3.0초 범위에서 보정한다.

### 3-6. 처리 흐름 다이어그램

```
[Android]
   │  binary frame 스트림 시작
   ▼
[Spring]   ─── 단순 릴레이 (메시지 크기 제한 관리)
   │
   ▼
[AI 서버]
   │
   ├─[즉시 검증]──── 첫 frame 도착 시점
   │     │
   │     ├─ 해상도 검사
   │     └─ frame decode 가능 여부
   │            │
   │            └─ failed면 즉시 응답, 이하 단계 진입 안 함
   │
   ▼
[2-3-2] 영상 전처리 + Frame-level 품질 검사 (블러/휘도/흔들림)
   │
   ▼
[2-3-3] MediaPipe Pose 추출
   │
   ▼
[2-3-4] 자세 지표 산출 (frame별 추론 결과는 즉시 응답)
   │
   ▼
[2-3-5] 측정 품질 판단 (visibility / 측면 구도 / 추적 안정성)
   │
   ▼
[분석 종료 trigger 발생]
   │   (사용자 명시적 종료 또는 마지막 frame 수신 후 2.0초 무수신)
   ▼
[누적 검증]──── 분석 종료 시점
   │
   ├─ 누적 분석 시간
   ├─ Effective FPS
   └─ 누적 frame 수
   │
   ├─ 3-2 우선순위 규칙 적용 (failed > low_confidence > success)
   │
   ▼
[2-3-6] 피드백 생성
   │
   ▼
[2-3-7] 최종 누적 결과 메시지 응답 ─── Spring ─── Android
```

## 4. 촬영 가이드 (단일 정답 위치)

본 장은 사용자에게 보여줄 촬영 가이드 메시지의 **유일한 정의 위치**이다. 다른 문서(2-3-2, 2-3-5) 및 프론트엔드 화면은 본 장의 메시지를 인용한다.

### 4-1. 분석 시작 전 가이드 (촬영 단계 안내)

사용자가 촬영을 시작하기 전 또는 촬영 중에 표시되는 가이드 메시지.

| 가이드 항목 | 사용자에게 보여줄 메시지 | 어떤 검증 항목과 연결되나 |
| --- | --- | --- |
| 촬영 환경 | 러닝머신 측면에서 카메라를 고정하고 촬영해주세요. | 본 문서 2장 (MVP 범위) |
| 분석 대상자 위치 | 분석 대상자가 화면 중앙에 가장 크게 보이도록 촬영해주세요. | 2-3-5 (분석 대상자 추적 안정성) |
| 전신 포함 | 머리부터 발끝까지 전신이 화면에 들어오도록 촬영해주세요. | 2-3-5 (visibility 검사) |
| 발 잘림 방지 | 발끝과 뒤꿈치가 화면 아래에서 잘리지 않도록 카메라를 조금 멀리 두세요. | 2-3-5 (foot visibility) |
| 측면 촬영 구도 | 몸의 옆면이 보이도록 카메라를 배치해주세요. | 2-3-5 (측면 구도 검사) |
| 카메라 고정 | 촬영 중 카메라가 흔들리지 않도록 고정해주세요. | 2-3-2 (Frame-level 흔들림), 2-3-5 (Stride-level 흔들림) |
| 러닝 시간 | 최소 10초 이상 연속으로 달려주세요. | 본 문서 3장 (누적 분석 시간 임계값) |

### 4-2. 분석 후 재촬영 안내 (실패 단계 메시지)

분석이 `failed` 또는 `low_confidence`로 종료되었을 때 사용자에게 표시되는 메시지는 reason code별로 다르며, **2-3-5 문서 8장의 Reason Code 표**에 정의된 사용자 메시지를 사용한다.

> **단일 정답 위치 원칙**: 사용자에게 보여줄 reason code별 메시지는 2-3-5에서만 정의한다. 본 문서는 촬영 시작 전 가이드 메시지만 정의한다. 이렇게 분리하는 이유는, 한 메시지가 두 문서에 있으면 표현이 불일치할 위험이 크기 때문이다.

## 5. 다른 모듈/파트와의 인터페이스

본 문서가 정의한 정책이 AI 파트 내 다른 문서와 다른 개발 파트(백엔드/프론트엔드)에서 어떻게 사용되는지 정리한다.

### 5-1. AI 파트 내부 인터페이스

| 본 문서가 정의 | 사용하는 문서 | 사용 방식 |
| --- | --- | --- |
| 누적 분석 시간 임계값 (5초/10초) | 2-3-2, 2-3-5 | "본 문서 3장의 임계값을 따른다"로 인용 |
| Effective FPS 임계값 (24/30fps) | 2-3-2의 fps 정규화 정책 | 정규화 입력 기준 |
| 해상도 임계값 (720p) | 2-3-2의 해상도 정규화 정책 | 정규화 입력 기준 (필요조건) |
| 상태 판정 우선순위 규칙 (`failed` > `low_confidence`) | 2-3-5 | 메타데이터 검증과 후속 검증 결과를 합칠 때 동일 규칙 적용 |
| MVP 범위 (측면/실내 러닝머신/실시간 스트림) | 2-3-2, 2-3-3, 2-3-4 | 모듈이 동작하는 가정 조건 |
| 분석 대상자 정책 (간섭 허용) | 2-3-5 (추적 안정성) | 추적 안정성 판단의 출발점 |
| 분석측 단일값 정책 | 2-3-4, 2-3-5, 2-3-7 | 자세 지표 계산 및 응답 메시지 단일값 응답 |
| 부분 결과 미제공 정책 | 2-3-5, 2-3-7 | 상태 분기 및 응답 구조 결정 |

### 5-2. 백엔드 인터페이스

| 본 문서가 정의 | 백엔드 처리 방식 | 비고 |
| --- | --- | --- |
| 통신 프로토콜 = WebSocket 실시간 스트림 | 클라이언트 ↔ Spring ↔ AI 서버 3-tier 릴레이 | 엔드포인트 `/ws/chobo-runner` |
| 입력 단위 = 단일 frame (binary) | Android에서 JPEG 압축된 binary frame을 단일 메시지로 전송 | 최대 메시지 크기 10MB |
| 메타데이터 임계값 검증 | **AI 서버에서 수행**, Spring은 단순 릴레이 | Spring은 메시지 크기 제한만 관리 |
| 분석 자동 종료 timeout (default 2.0초) | Spring/AI 서버 heartbeat 정책과 동기화 필요 | heartbeat interval은 본 timeout보다 짧아야 함 |
| 분석 결과 응답 단위 | frame별 추론 결과 JSON 즉시 응답 + 최종 누적 결과 응답 | 메시지 스펙은 2-3-7 |
| 분석측 단일값 응답 | `metrics`는 단일값, `analysis_side` 필드 포함 | 응답 메시지 스펙은 2-3-7 |
| 부분 결과 미제공 정책 | 최종 누적 결과 메시지는 핵심 지표 3개 묶음으로만 응답 | frame별 즉시 응답은 별도 |

> **백엔드 설계 문서와의 관계**: WebSocket 연결 수립 절차, 세션 매핑 방식, 메시지 크기 제한, heartbeat, 재연결 전략 등 통신 인프라 측면은 백엔드 WebSocket 설계 문서에서 정의된다. 본 문서는 통신 위에서 흐르는 데이터의 의미(어떤 frame이 분석 대상으로 적합한가)만 정의한다.
> 

### 5-3. 프론트엔드 인터페이스

| 본 문서가 정의 | 프론트엔드 처리 방식 |
| --- | --- |
| 촬영 가이드 메시지 (4-1장) | 촬영 시작 전 화면 UI에 가이드 문구로 표시 |
| 입력 단위 = 실시간 스트림 | 사용자가 카메라 권한 부여 후 실시간 분석 화면 진입, frame 자동 전송 |
| JPEG 압축 | 백엔드 메시지 크기 제한(10MB)을 위해 Android에서 frame 압축 후 전송 |
| 부분 결과 미제공 정책 | 결과 화면이 항상 핵심 지표 3개를 모두 표시한다고 가정 가능 |
| 분석측 표시 | 결과 화면에 *"오른쪽 다리 기준"* 또는 *"왼쪽 다리 기준"* 표기 (분석측 결정은 AI 서버에서 자동 수행) |
| 분석 상태값 | "분석 중" / "분석 완료" / "재촬영 필요" 화면 상태 매핑 (상태값 정의는 2-3-5) |

## 6. 메타데이터 임계값 보정 실험 계획

본 문서가 정의한 임계값의 타당성을 실측 데이터로 검증하기 위한 실험. 이 실험은 데모 영상 수집 후 진행한다.

| 실험 항목 | 방법 | 검증 대상 |
| --- | --- | --- |
| 누적 분석 시간별 IC 후보 수 | 5초/7초/10초 영상 비교 | "5초 미만 = failed" 기준이 적절한가 |
| 5~10초 구간 결과 안정성 | 중앙값 및 사분위 범위(IQR) 비교 | "5~10초 = low_confidence" 처리가 적절한가 |
| 24fps와 30fps의 IC 검출 정확도 | 동일 영상의 fps만 변환하여 비교 | "24fps 미만 = failed" 기준이 적절한가 |
| 720p와 1080p의 검출률 | 해상도별 landmark visibility 평균 비교 | "720p 권장" 기준이 적절한가 |
| **네트워크 지연 환경에서 Effective FPS 변화** | LTE / 5G / Wi-Fi 환경에서 nominal 30fps 대비 effective fps 측정 | "Effective FPS 24 미만 = failed" 기준이 실환경에서 적절한가, default timeout 2.0초가 충분한가 |

> 자세 지표(Foot Strike, Knee Flexion, Trunk Lean) 자체의 분류 임계값 검증 실험은 2-3-4 문서의 검증 계획을 따른다. JPEG 압축 품질에 따른 landmark 안정성 실험은 2-3-2 문서의 검증 계획에서 다룬다. 본 문서는 영상 메타데이터 임계값에만 집중한다.
>

## 7. 최종 정리

본 문서는 AI 파트의 입력 영상 규격과 사용자 촬영 가이드의 유일한 정의 위치이다.

핵심 결정 사항을 다시 정리하면 다음과 같다.

1. MVP 입력 범위는 **러닝머신에서 측면 고정 촬영한 영상**으로 제한하며, 입력은 **WebSocket 기반 실시간 스트림** 단일 모드로 처리한다.
2. 입력 처리 구조는 **Android → Spring → AI 서버 3-tier 릴레이**이며, 메타데이터 검증은 AI 서버에서 수행한다.
3. 영상 메타데이터 임계값은 **누적 분석 시간 5초 이상(failed 기준), 10초 이상(권장), Effective FPS 24 이상, 해상도 720p 이상**으로 한다. FPS는 nominal이 아닌 **effective** 기준이다.
4. 메타데이터 검증은 시점에 따라 **즉시 검증** (해상도, decode 가능 여부)과 **누적 검증** (분석 시간, effective fps, frame 수)으로 분리된다.
5. 검사 결과 충돌 시 **`failed` > `low_confidence` > `success`** 우선순위로 최종 상태를 결정한다.
6. 분석 자동 종료는 **마지막 frame 수신 후 2.0초** (default) 무수신을 기준으로 하며, 백엔드 heartbeat 정책과 동기화한다.
7. 해상도 기준은 **필요조건**이며, "사람이 충분히 크게 보이는가"의 충분조건은 2-3-5에서 검증한다.
8. 분석 대상자 정책은 **단일 사용자 권장**이되, 다중 인물이 보인다는 이유만으로 즉시 거부하지 않으며, 추적 안정성 기반 판단은 2-3-5 문서를 따른다.
9. 분석측은 **한쪽 다리만 단일값으로 측정**하며, 좌우 분리 측정은 하지 않는다.
10. 부분 결과는 1차 MVP에서 제공하지 않으며, **핵심 지표 3개 모두 계산 가능 시에만** 분석 결과를 제공한다.
11. 사용자 촬영 가이드 메시지는 본 문서 4장을 단일 정답 위치로 사용한다.

품질 검사 알고리즘, 지표 계산식, reason code 정의, 응답 메시지 스펙 등 모든 후속 처리는 각 담당 문서에 정의되어 있으며, 본 문서는 그쪽으로 참조하는 역할만 수행한다.
