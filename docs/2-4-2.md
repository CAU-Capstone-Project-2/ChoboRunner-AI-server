2-4-2. AI ↔ Backend WebSocket 연동 설계

> **작성·소유: 백엔드 (재민).** 파트 간 인터페이스 문서로, binary frame wire format·`stop` 신호 등 transport 규약의 단일 정답이다. AI 서버는 이 규약을 수신·구현하는 입장이며, 응답 메시지 본문 스펙은 docs/2-3-7이 단일 정답이다.

## 1. 이 문서를 한눈에

### 이 문서가 정의하는 것

이 문서는 **"Android → Spring → AI 서버로 영상 frame이 전달되고 분석 결과가 돌아오는 WebSocket 연동의 전체 스펙"** 을 정의합니다. 구체적으로:

- **연결 토폴로지**와 AI 서버가 노출하는 WebSocket 엔드포인트
- **binary frame wire format** — capture timestamp 8 byte 헤더 + JPEG (백엔드 재민 확정 규약)
- **제어 메시지**(분석 종료 신호 등)와 **응답 메시지**의 송수신 흐름
- AI 서버 `server/` 모듈의 **WebSocket 레이어 구현 설계** (세션 생명주기, 동시성, 백프레셔)
- 8 byte timestamp 헤더의 **AI 서버 측 파싱·fallback 처리**

### 이 문서가 정의하지 않는 것

| 알고 싶은 것                                                                                          | 가야 할 문서                                                                       |
| ----------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------- |
| 응답 메시지 4종(`frame_inference` / `analysis_progress` / `analysis_result` / `error`)의 필드 스펙은? | **2-3-7** (응답 메시지 단일 정답)                                                  |
| capture timestamp 누락 시 fallback·`timestamp_fallback` 플래그 정책은?                                | **2-3-2** (§3-3 timestamp 처리)                                                    |
| Frame-level 품질 검사(휘도, 블러, 흔들림)는?                                                          | **2-3-2** (4장)                                                                    |
| Pose 후 품질 게이트·상태값(success/low_confidence/failed) 결정은?                                     | **2-3-5**                                                                          |
| 자세 지표 계산식·IC 검출은?                                                                           | **2-3-4**                                                                          |
| Spring relay 내부 구현(`VideoRelayHandler.java`), Android 송신 코드는?                                | **백엔드 레포 `ChoboRunner-Backend`** (본 문서 §4는 그 규약을 AI 관점에서 옮긴 것) |

### 다른 파트가 알아두면 좋은 핵심 결정 5가지

1. **timestamp 책임은 Android에 있다.** 각 binary frame은 `[8B capture timestamp][JPEG]` 형태로 송신되며, AI 서버는 이 timestamp로 IC 검출의 시간축을 잡는다. AI 서버가 timestamp를 임의 생성하는 것은 **fallback 경로**(정확도 손실)다.
2. **1 binary message = 1 frame.** 헤더와 JPEG는 반드시 같은 WebSocket frame에 담는다. frame 묶음 송신·헤더 분리 송신 금지.
3. **분석 시작은 별도 핸드셰이크 없이 첫 frame 수신 시점이며, 종료는 text frame `{"type":"stop"}`로 신호한다.** 별도 `start` 메시지는 없다 — Spring이 보내는 첫 binary frame이 곧 분석 시작이다. frame 송신 중단(연결 끊김)·무수신 타임아웃만으로도 AI 서버는 종료로 간주하지만, 정상 종료는 `stop` 송신을 권장한다.
4. **AI 서버의 분석은 stateful 세션이다.** 연결 1개 = 분석 세션 1개 = `Pipeline` 인스턴스 1개. 현재 `pipeline.py`는 file path 배치 모드(`run_on_video_file`) 전용이므로, 본 연동을 위해 **증분(stream) 진입점 추가가 선행**되어야 한다 (§7-3).
5. **응답 메시지는 2-3-7의 4종 그대로.** 본 문서는 transport만 정의하고, 메시지 본문 스펙은 2-3-7이 단일 정답이다.

## 2. 연결 토폴로지

```
[Android]  ──(WebSocket)──▶  [Spring relay]  ──(WebSocket)──▶  [AI 서버]
   카메라 frame              VideoRelayHandler          server/routes/stream.py
   capture ts 부여           길이 검증·로깅·그대로 relay   8B 헤더 파싱 + 분석
```

- Android와 AI 서버는 **직접 연결되지 않는다.** Spring이 중계(relay)한다.
- Spring은 binary frame의 **payload(8B 헤더 포함)를 변형 없이 그대로** AI 서버로 전달한다. 즉 AI 서버가 수신하는 binary frame도 `[8B ts_ms][JPEG]` 형태다.
- Spring은 길이 `< 8 byte` binary frame을 **drop**한다 (AI 서버까지 도달하지 않음). 따라서 AI 서버가 받는 binary frame은 `≥ 8 byte`로 가정 가능하나, 방어적 파싱은 유지한다 (§8-1).

### 연결 정보

| 항목 | 값 |
| --- | --- |
| Spring 엔드포인트 (Android용) | `wss://capstone2-backend.foopky.com/ws/chobo-runner` (운영) / `ws://<host>:8080/ws/chobo-runner` (개발) |
| AI 서버 엔드포인트 (Spring용) | `WS /ws/inference` — `server/routes/stream.py` |
| Subprotocol | 사용 안 함 |
| Frame size 한도 | 10 MB (`app.websocket.max-message-size-bytes=10485760`) |

## 3. 메시지 흐름

WebSocket 한 연결에서 **binary frame**(영상)과 **text frame**(제어·응답 JSON)을 혼용한다.

```
Spring ──▶ AI:   <binary: [8B ts_ms][JPEG]>   <binary>  ...      (영상 frame, ~30fps)
Spring ──▶ AI:   {"type":"stop"}                                  (text, 분석 종료 신호)

AI ──▶ Spring:   {"type":"frame_inference",  ...}                 매 frame 추론 직후
AI ──▶ Spring:   {"type":"analysis_progress", ...}                단계 전환 / stride 갱신 시
AI ──▶ Spring:   {"type":"analysis_result",  ...}                 stop 수신(또는 연결 종료) 시 1회
AI ──▶ Spring:   {"type":"error", ...}                            시스템 처리 실패 시
```

이 흐름은 docs/2-3-7 §2 "메시지 흐름 예시"와 동일하다. 본 문서는 그 흐름을 WebSocket transport 위에 매핑한 것이다.

> **별도 세션 시작 핸드셰이크는 없다.** Spring이 보내는 첫 binary frame이 곧 분석 시작 시점이다. `analysis_side`·`direction`은 WebSocket으로 전달되지 않으며, v1에서 AI 서버는 기본값(`analysis_side="left"`, `direction="left_to_right"`)을 사용한다 (pipeline.py 현행 default와 동일). 영상에서의 자동 결정은 docs §3-1 Phase 9 anchor.

## 4. Binary frame wire format (백엔드 확정 규약)

> 본 절은 백엔드(재민)가 확정한 규약이며 단일 정답은 백엔드 레포다. AI 서버는 이 규약을 **수신·파싱**하는 입장에서 옮겨 적는다.

### 4-1. 레이아웃

**Android 송신 1개 binary frame = `[8B ts_ms 헤더] + [JPEG bytes]`**

| Offset | Size | Field   | Type         | Encoding                        |
| ------ | ---- | ------- | ------------ | ------------------------------- |
| 0      | 8    | `ts_ms` | signed int64 | Big-endian (network byte order) |
| 8      | N    | `jpeg`  | JPEG image   | JPEG bytes (`FFD8` … `FFD9`)    |

- 한 binary WebSocket frame = `8 + N` bytes.
- 헤더와 JPEG는 **반드시 같은 frame**에 담는다. 분리 송신 금지.
- JPEG는 카메라 인코더 결과물 그대로. EXIF·메타데이터 추가 금지.

### 4-2. `ts_ms` 필드 상세

| 항목       | 값                                                                              |
| ---------- | ------------------------------------------------------------------------------- |
| 단위       | 밀리초 (ms)                                                                     |
| 크기       | 8 byte = 64-bit signed long                                                     |
| Byte order | Big-endian (Java / 네트워크 표준)                                               |
| 시계       | 단조(monotonic) 시계 — Android `SystemClock.elapsedRealtimeNanos() / 1_000_000` |
| 유효 범위  | 양수. `0` 또는 음수는 **fallback 신호**로 해석                                  |
| 단조성     | 같은 세션 내 `ts_(n+1) >= ts_n` (단조 비감소). 단조 시계 사용으로 자동 보장     |

- **8 byte인 이유**: 4 byte unsigned ms는 약 49일까지만 표현 가능. `elapsedRealtimeNanos`는 부팅 후 며칠이면 49일 표현 한계를 넘긴다. 8 byte signed long으로 안전 확보.
- **epoch 가정 금지**: `ts_ms`는 부팅 기준 단조 시계이지 wall-clock(Unix epoch)이 아니다. AI 서버는 절대 시각이 아니라 **frame 간 간격(Δt)** 만 사용한다 (IC 검출의 시간축).
- **drop된 frame의 ts 보정 금지**: 네트워크 사정으로 frame을 drop하는 것은 허용되나, drop된 frame 자리를 임의 ts로 채우지 않는다. 실제 캡처된 frame의 실제 ts만 보낸다.

### 4-3. Spring relay 동작 매트릭스

| 상황                    | Spring 동작                                                               |
| ----------------------- | ------------------------------------------------------------------------- |
| binary frame 길이 `< 8` | **drop**. AI 서버까지 가지 않음. Spring 로그에 경고                       |
| binary frame 길이 `≥ 8` | 헤더 파싱(로깅용) 후 **payload 전체(헤더 포함)를 AI 서버로 그대로 relay** |
| `ts_ms = 0` 또는 음수   | drop ❌ — 그대로 relay. **AI 서버가 fallback 처리** (§8-2)                |
| 헤더 누락한 raw JPEG    | AI 서버에서 `error` 응답 (잘못된 byte로 JPEG decode 시도 → 실패)          |

## 5. 제어·응답 메시지 (text frame)

### 5-1. 수신 — 제어 메시지

현재 확정된 수신 제어 메시지는 **`stop` 1종**이다.

```json
{ "type": "stop" }
```

- 발생 시점: 촬영 세션 종료 시 Android가 송신, Spring이 relay.
- AI 서버 동작: 현재까지 누적된 결과로 `analysis_result`를 1회 조립·송신한 뒤 세션을 정리한다.
- `stop` 없이 연결이 끊긴 경우(`WebSocketDisconnect`)도 동일하게 종료로 간주하고 `analysis_result` 송신을 시도한다 (전송 실패는 swallow).

`server/schemas.py`에 수신 제어 메시지용 Pydantic 모델을 둔다 (응답 4종은 `result_serializer.py` 재사용 — 중복 정의 금지):

```python
# server/schemas.py (예상 스켈레톤)
from typing import Literal
from pydantic import BaseModel

class StopMessage(BaseModel):
    type: Literal["stop"]
```

### 5-2. 송신 — 응답 메시지

응답은 모두 **text frame (JSON)** 이며, `result_serializer.py`의 Pydantic 모델 4종을 `.model_dump_json()`으로 직렬화해 그대로 보낸다.

| `type`              | 모델 (`result_serializer.py`) | 송신 시점                                 |
| ------------------- | ----------------------------- | ----------------------------------------- |
| `frame_inference`   | `FrameInferenceMessage`       | 매 frame 추론 직후                        |
| `analysis_progress` | `AnalysisProgressMessage`     | 단계 전환 / stride 갱신 시                |
| `analysis_result`   | `AnalysisResultMessage`       | `stop` 수신 또는 연결 종료 시 1회         |
| `error`             | `ErrorMessage`                | 시스템 처리 실패 (decode 실패, 내부 예외) |

필드 스펙은 docs/2-3-7이 단일 정답. 본 문서는 송신 transport와 시점만 정의한다.

## 6. error vs failed 경계

docs/2-3-7 §3-4 정책을 WebSocket 맥락에서 재확인한다.

| 상황                                                        | 메시지                                                      |
| ----------------------------------------------------------- | ----------------------------------------------------------- |
| JPEG decode 실패, 헤더 누락 raw JPEG, 내부 예외             | `error` (`ErrorMessage`) — **시스템 에러**                  |
| 품질 미달로 분석 불가 (pose 미검출 누적, 측면 구도 이탈 등) | `analysis_result`의 `status: "failed"` — **비즈니스 결과**  |
| 단일 frame 비정상 (pose 미검출 1장, 품질 플래그)            | `frame_inference`의 `result` 내부로 표현 (별도 status 없음) |

`failed_to_decode_binary_image` 같은 디코딩 실패는 `ErrorMessage.error_code`로 표현한다.

## 7. AI 서버 측 구현 설계 (`server/` 모듈)

### 7-1. 파일 구성

```
server/
├── main.py            FastAPI app + lifespan(AppConfig·모델 1회 로드) + 라우터 등록
├── schemas.py         수신 제어 메시지 Pydantic (stop 등). 응답 4종은 result_serializer 재사용
├── session.py         StreamSession — 연결 1개의 상태 컨테이너            ★신규
└── routes/
    ├── analyze.py      POST /analyze — run_on_video_file 래핑 (배치 모드, 우선 구현)
    └── stream.py       WS /ws/inference — 얇은 I/O 루프, 분석 로직 0줄
```

CLAUDE.md §6 정책(로직/오케스트레이션 분리)에 따라 `stream.py`는 **I/O만** 담당한다: WebSocket 수신, 8B 헤더 파싱, `StreamSession` 위임, 응답 JSON 송신, 예외 → `ErrorMessage`. 분석 로직은 전부 `pipeline.py`/단위 모듈.

### 7-2. `StreamSession` — 연결당 상태

연결 1개 = `StreamSession` 1개. 내부에 stream용 `Pipeline`(§7-3)을 소유하고, 연결 종료 시 반드시 정리한다.

```python
# server/session.py (예상 스켈레톤)
class StreamSession:
    def __init__(self, cfg: AppConfig, analysis_side, direction): ...
    async def on_binary_frame(self, raw: bytes) -> FrameInferenceMessage: ...
    async def maybe_progress(self) -> AnalysisProgressMessage | None: ...
    async def finalize(self) -> AnalysisResultMessage: ...
    def close(self) -> None: ...   # Pipeline.close() — PoseExtractor 누수 방지
```

### 7-3. `pipeline.py` 선행 리팩터링 — 증분 진입점

현재 `run_on_video_file`은 frame loop와 누적 평가(§2~§9 블록)가 한 함수에 묶여 있어 스트림에서 재사용할 수 없다. 다음이 선행되어야 한다:

1. **누적 블록 추출**: `run_on_video_file`의 §2~§9(reason code 누적 → metrics → `build_analysis_result`)를 `_accumulate(landmarks_series, ...) -> PipelineResult` 공용 함수로 분리. 배치/스트림이 동일 로직 공유.
2. **stream 진입점 추가**: `StreamPipeline`(또는 `Pipeline`의 메서드 세트).
   - `push_frame(frame, timestamp_ms)` — 기존 `_extract_and_evaluate_one_frame` 재사용 + `landmarks_series` 증분 누적 → `FrameInferenceMessage` 산출.
   - `snapshot_progress()` — 지금까지 누적분으로 IC/metrics 부분 재계산 → `AnalysisProgressMessage`. **매 frame이 아니라 N frame마다/단계 전환 시**만 호출 (IC 재계산 비용).
   - `finalize()` — `_accumulate` 호출 → `AnalysisResultMessage`.

> 이 리팩터링은 별도 Phase로 분리해 anchor를 박는다. pipeline.py 모듈 docstring(현재 "file path mode 전용")도 함께 갱신한다.

### 7-4. 동시성·스레드 모델

- MediaPipe `PoseLandmarker`는 thread-safe가 아니고 추론은 CPU 바운드다. → **연결 1개당 `Pipeline` 1개**, 추론은 `await loop.run_in_executor(...)`로 이벤트 루프 밖에서 실행.
- `AppConfig`와 `.task` 모델 파일은 FastAPI `lifespan`에서 **1회 로드**해 공유. 연결마다 재로드 금지. (단 `Pipeline.__init__`이 `PoseExtractor`를 새로 생성하므로 다중 연결 시 초기화 비용 존재 — 필요 시 풀링 검토.)

### 7-5. 백프레셔

추론 속도가 frame 도착 속도(~30fps)보다 느리면 큐가 쌓인다.

- 수신 루프는 **최신 frame만 유지**하고 밀린 frame은 drop한다 (실시간 분석은 최신성 우선).
- drop은 `frame_index` 갭으로 자연 표현된다. ts는 보정하지 않는다 (§4-2).

### 7-6. 세션 생명주기·정리

- 연결 수립 → `StreamSession` 생성, `Pipeline.__enter__`. 첫 binary frame 수신 시 분석 시작 (별도 시작 메시지 없음).
- 종료 트리거 3종:
  - (a) 정상 — `{"type":"stop"}` text frame 수신
  - (b) `WebSocketDisconnect` — 연결 끊김
  - (c) **무수신 타임아웃 — 마지막 frame 수신 후 3초 경과** (§9 결정)
- 종료 시 → `finalize()` 후 `analysis_result` 송신 시도 → **반드시 `Pipeline.close()`** (`PoseExtractor` 누수 방지). `try/finally`로 보장.

### 7-7. 서버 실행

프로젝트 루트(`ChoboRunner-AI-server/`)에서 uvicorn으로 `server.main:app`을 띄운다.

```powershell
# 개발용 (코드 변경 자동 리로드)
uvicorn server.main:app --reload --port 8000

# Backend 연동용 (외부 접속 허용)
uvicorn server.main:app --host 0.0.0.0 --port 8000
```

| 항목 | 값 | 출처 |
| --- | --- | --- |
| WebSocket 엔드포인트 | `ws://<host>:8000/ws/inference` | `WebSocketConfig.endpoint_path` (§9 #2) |
| 무수신 타임아웃 | 3.0초 | `WebSocketConfig.no_frame_timeout_sec` (§7-6 (c) / §9 #4) |
| binary frame 크기 한도 | 10 MB | `WebSocketConfig.max_message_bytes` (§2 연결 정보) |

엔드포인트 경로는 `server/main.py`가 `cfg.websocket.endpoint_path`로 등록한다 — 경로 변경은 uvicorn 인자가 아니라 `config.py`의 `WebSocketConfig`(또는 환경변수)에서 한다. `--host`/`--port`는 transport 파라미터이므로 실행 시 지정한다.

## 8. Timestamp 처리 (AI 서버 측)

### 8-1. 8B 헤더 파싱

`stream.py`가 binary frame을 받으면:

```python
import struct, time

TIMESTAMP_HEADER_BYTES = 8

def parse_binary_frame(raw: bytes) -> tuple[float | None, bytes]:
    """[8B BE int64 ts_ms][JPEG] → (capture_ts_sec | None, jpeg_bytes)."""
    if len(raw) < TIMESTAMP_HEADER_BYTES:
        raise ValueError("binary frame too short (< 8B header)")
    ts_ms = struct.unpack(">q", raw[:TIMESTAMP_HEADER_BYTES])[0]   # BE signed int64
    jpeg_bytes = raw[TIMESTAMP_HEADER_BYTES:]
    capture_ts = ts_ms / 1000.0 if ts_ms > 0 else None             # 0/음수 → fallback
    return capture_ts, jpeg_bytes
```

파싱된 `(capture_ts, jpeg_bytes)`는 `video_preprocessor`의 streaming 진입점에 그대로 연결된다 — 이 모듈은 이미 스트림 입력을 전제로 설계되어 있다:

- `decode_jpeg_binary(jpeg_bytes)` — JPEG binary → BGR `np.ndarray`, 실패 시 `None`.
- `resolve_timestamp(capture_ts, server_recv_ts)` — `capture_ts`가 `None`이면 `server_recv_ts`로 fallback + `is_fallback=True`.
- `process_binary_frame(jpeg_bytes, capture_ts, server_recv_ts)` — 위 둘을 조립.

즉 `ts_ms <= 0`이면 `capture_ts=None`으로 넘기는 것만으로 기존 fallback 경로(docs/2-3-2 §3-3)가 자동 동작한다.

### 8-2. fallback 정책

| 입력                    | AI 서버 처리                                                                                                                     |
| ----------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| `ts_ms > 0` (정상 경로) | `capture_ts = ts_ms/1000` 사용. `timestamp_fallback` 플래그 없음                                                                 |
| `ts_ms <= 0`            | `capture_ts = None` → 서버 수신 시각 사용 + `quality_flags`에 `timestamp_fallback` 추가 (`FrameQualityFlag` Literal에 이미 존재) |

누적 `timestamp_fallback` 비율이 `FramePreprocessConfig.timestamp_fallback_ratio_max`(default 0.3)를 초과하면 docs/2-3-5 품질 게이트로 신호가 전달된다 (시간 정확도 저하).

### 8-3. MediaPipe stream 모드 단조성 가드

MediaPipe `PoseLandmarker`의 stream 모드 `process_frame(frame, timestamp_ms)`는 **timestamp가 엄격히 증가(strictly increasing)** 해야 한다. 그러나 §4-2의 wire 규약은 **비감소(non-decreasing)** 만 보장하므로, 동일 `ts_ms`를 가진 두 frame이 연속될 수 있다.

→ stream 레이어는 MediaPipe에 넘기는 timestamp가 직전 값보다 같거나 작으면 **`+1 ms` bump**해 엄격 증가를 보장한다. (이 bump는 MediaPipe 호출용 내부 값일 뿐, 분석에 쓰는 `capture_ts`는 원본 유지.)
