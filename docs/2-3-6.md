2-3-6. Rule-based 실시간 피드백

## 1. 이 문서를 한눈에

### 이 문서가 정의하는 것

이 문서는 **"누적된 자세 분석 통계를 사용자에게 보여줄 음성/텍스트 피드백 메시지로 변환하는 룰"** 을 정의합니다. AI 파트 안에서 이 문서는 다음 세 가지의 **단일 정답 위치(single source-of-truth)** 역할을 합니다.

- **피드백 트리거 룰** (어떤 분석 결과가 어떤 메시지로 변환되는가)
- **피드백 메시지 사전** (실제 사용자에게 보여줄 음성용/화면용 문구)
- **피드백 출력 정책** (status별 출력, 우선순위, 빈도 제한, 출력 한도)

### 이 문서가 정의하지 않는 것

다음 항목은 다른 문서에 정의되어 있으니 그쪽을 참조해주세요.

| 알고 싶은 것 | 가야 할 문서 |
| --- | --- |
| 자세 지표(Foot Strike, Knee Flexion, Trunk Lean)는 어떻게 계산되나요? | **2-3-4** (자세 지표 산출) |
| 분석 결과의 success/low_confidence/failed 상태값은? | **2-3-5** (6장 분기 룰) |
| Reason code 사용자 메시지(예: 재촬영 안내)는? | **2-3-5** (8장 reason code 사전) |
| 응답 메시지 JSON 스펙은? | **2-3-7** (자세 분석 결과 구조화) |
| TTS(Text-to-Speech) 엔진 선택, 음성 합성 자체는? | Android 구현 영역 (본 문서 책임 외) |

### 다른 파트가 알아두면 좋은 핵심 결정 7가지

이 문서에서 결정된 사항 중, 다른 파트(AI 모듈, Android)에 영향을 주는 핵심 정책은 다음과 같습니다.

1. **본 모듈은 누적 통계 기반 피드백만 생성한다.** 단일 stride 기반 즉시 피드백은 false alarm을 유발하므로 사용하지 않는다(2-3-4 9-5절).
2. **첫 음성 피드백 출력은 분석 시작 후 약 2~3초 뒤를 목표로 한다.** 누적 stride 3개 + IC 검출/누적/처리 지연을 포함한 실측 범위.
3. **`success` / `low_confidence` / `failed` 상태별 피드백 정책이 다르다.** `failed`에서는 자세 피드백을 제공하지 않으며, `low_confidence`에서는 신뢰도 안내를 함께 제공한다.
4. **TTS는 cycle당 최대 1개, 화면은 최대 3개**까지 출력한다. 여러 메시지가 동시에 트리거되어도 음성은 최우선 1개만 합성된다.
5. **메시지는 `tts_text`(음성용 짧은 문구)와 `display_text`(화면용 자세한 문구)로 분리된다.** 음성은 러닝 중 듣기 쉽도록 짧게, 화면은 정보량을 더 담는다.
6. **Foot Strike Pattern은 기본 TTS False**로 출력하며, 화면과 최종 결과에 표시한다. 정보성 지표라 음성 반복 시 사용자 피로감이 크기 때문.
7. **본 모듈의 출력은 텍스트 메시지다.** 실제 음성 합성(TTS)은 Android에서 수행한다. 본 모듈은 *"무슨 말을 할지"* 만 결정하고, *"어떻게 말할지"* 는 Android의 책임이다.

## 2. 본 단계의 위치

본 단계는 **자세 지표 산출(2-3-4) 직후, 응답 메시지 생성(2-3-7) 직전**에 동작한다.

```
[2-3-4 자세 지표 산출 + stride 누적 통계]
     │ stride 누적 통계 (median, IQR, 분류 라벨)
     ▼
[2-3-5 측정 품질 판단]
     │ status (success/low_confidence/failed) + reason codes
     ▼
┌─────────────────────────────────────┐
│ 본 문서 담당                          │
│  ├ status별 피드백 출력 정책 적용    │
│  ├ 누적 통계 → 피드백 메시지 변환 룰 │
│  ├ 우선순위 평가 (어느 메시지 먼저?) │
│  ├ 출력 한도 적용 (TTS 1개, 화면 3개)│
│  ├ 중복 방지 (5초 내 같은 메시지 X)  │
│  └ 피드백 메시지 텍스트 출력         │
│      (tts_text + display_text)      │
└─────────────────────────────────────┘
     │ 피드백 메시지 (텍스트)
     ▼
[2-3-7 응답 메시지 구조화]
     │ JSON 응답
     ▼
[Android]
     │ TTS 엔진으로 음성 합성 (tts_enabled True인 메시지만)
     ▼
[사용자에게 음성 출력 + 화면 표시]
```

### 본 단계의 책임 경계

| 본 단계 (2-3-6) | 다른 단계 |
| --- | --- |
| 자세 분석 통계 → 피드백 메시지 변환 | — |
| 메시지 우선순위, 빈도 제한, 출력 한도 | — |
| 가치 판단 회피 원칙 적용 | — |
| 음성용/화면용 문구 분리 | — |
| 자세 지표 계산 | **2-3-4** |
| 분석 상태값 결정 | **2-3-5** |
| 응답 메시지 JSON 직렬화 | **2-3-7** |
| 음성 합성 (TTS) | **Android** (본 문서 책임 외) |
| 사용자 화면 표시 스타일 | **Android** (본 문서 책임 외) |

## 3. 피드백 출력 정책

### 3-1. 피드백 출력 시점

본 모듈의 피드백 출력은 **누적 stride 3개 이상이 모인 후**부터 시작된다(2-3-4의 누적 정책과 동기화).

| 시점 | 본 모듈 동작 |
| --- | --- |
| 분석 시작 ~ 첫 IC 검출 전 | 피드백 출력 없음 |
| 첫 IC 검출 ~ 유효 stride 2개 누적 | 피드백 출력 없음 |
| 유효 stride 3개 누적 후 | 첫 피드백 출력 시작 |
| stride가 추가될 때마다 | 누적 통계 갱신 후 트리거 룰 재평가 |

> **첫 피드백 출력까지의 지연**: 평균 stride time 0.7초 기준 3 stride 누적은 약 2.1초가 소요되며, IC 검출 / 누적 / 처리 지연까지 포함하면 **약 2~3초 범위가 실측 목표**이다. 이론적 최단 지연(1.5초)은 모든 단계가 최선의 경우이므로, 운영상 **2~3초 목표 범위**로 설계한다.
> 

### 3-2. 피드백 빈도 제한 (중복 방지)

| 정책 | 내용 |
| --- | --- |
| 동일 메시지 중복 방지 | 같은 트리거 룰의 메시지는 **5초 내 재출력 안 함** |
| 다른 메시지 간 최소 간격 | 서로 다른 메시지 간 **2초 최소 간격** |
| 긍정 메시지 빈도 | "현재 자세가 안정적입니다" 등은 **30초 이상 간격** |
| 우선순위 충돌 시 | 우선순위 높은 메시지가 출력되며, 낮은 메시지는 다음 cycle로 연기 |

> **피드백 fatigue 방지의 중요성**: 매 stride마다 같은 *"무릎 굴곡이 부족합니다"* 메시지가 0.7초마다 반복 출력되면, 사용자는 곧 음성을 무시하게 된다. 본 빈도 제한은 사용자가 메시지에 집중할 수 있는 환경을 유지하기 위함이다.
> 

### 3-3. 피드백 우선순위

여러 트리거 룰이 동시에 활성화될 수 있다. 다음 우선순위로 출력 메시지를 결정한다.

| 우선순위 | 카테고리 | 비고 |
| --- | --- | --- |
| 1 (최우선) | 시스템 안내 (재촬영 등) | 2-3-5의 reason code 메시지 — 본 문서가 직접 생성하지 않고 인용 |
| 2 | 자세 경고 (Above Typical Range / Below Typical Range) | 본 문서가 정의하는 트리거 룰 |
| 3 | 자세 안내 (Typical Range — 긍정 메시지) | 빈도를 낮추어 가끔만 출력 |

> **시스템 안내가 최우선인 이유**: 분석이 진행되지 않을 정도로 품질이 낮은 경우, 자세 피드백을 보내는 것은 의미가 없다. *"카메라가 흔들리고 있습니다"* 가 *"무릎을 더 굽혀보세요"* 보다 먼저 사용자에게 도달해야 한다.
> 

### 3-4. status별 피드백 출력 정책

2-3-5가 결정한 분석 상태값(`success` / `low_confidence` / `failed`)에 따라 본 모듈의 출력 정책이 달라진다.

| status | 자세 피드백 출력 | 시스템 안내 | 비고 |
| --- | --- | --- | --- |
| `success` | 정상 출력 (모든 트리거 룰 평가) | 해당 없음 | 일반적인 분석 완료 상태 |
| `low_confidence` | **제한적 출력** + 신뢰도 안내 prefix | 우선 출력 | 자세 피드백을 제공하되 *"참고용으로만 확인해주세요"* 안내 동반 |
| `failed` | **출력 안 함** | reason code 메시지만 출력 | 자세 피드백 없이 시스템 안내만 |

### `low_confidence`에서의 신뢰도 안내 prefix

`low_confidence` 상태에서 자세 피드백을 출력할 때, 메시지 앞에 신뢰도 안내가 prefix로 붙는다.

**예시**:

| status | 출력 메시지 |
| --- | --- |
| `success` | "착지 시 무릎 굴곡이 일반 범위보다 작게 관찰됩니다." |
| `low_confidence` | "**측정 신뢰도가 낮아 참고용으로만 확인해주세요.** 착지 시 무릎 굴곡이 일반 범위보다 작게 관찰됩니다." |

> **`low_confidence`에서 자세 피드백을 완전히 막지 않는 이유**: 사용자 입장에서 분석을 끝까지 수행한 후 결과를 못 보는 경험은 부정적이다. 신뢰도가 낮음을 명시하면서 결과를 함께 제공하는 것이, 결과를 차단하는 것보다 사용자 경험이 좋다. 다만 사용자가 결과를 과신하지 않도록 prefix가 분명히 표기된다.
> 

> **`failed`에서 자세 피드백을 차단하는 이유**: `failed`는 핵심 지표 중 하나 이상이 계산 불가 상태이거나 심각한 품질 문제가 있는 상태다(2-3-5 6-1절). 이 상태에서 잘못된 자세 정보를 사용자에게 보여주면, 실제로는 측정이 실패한 영상에 기반한 정보를 신뢰하게 만들 위험이 있다.
> 

### 3-5. 출력 한도 (TTS 1개, 화면 3개)

여러 트리거 룰이 동시에 활성화되어도 사용자에게 한 번에 출력되는 메시지 수는 다음과 같이 제한된다.

| 출력 채널 | cycle당 최대 출력 | 선택 기준 |
| --- | --- | --- |
| **TTS (음성)** | **1개** | 우선순위 가장 높은 메시지만 음성 합성 |
| **화면 (시각 UI)** | **최대 3개** | 우선순위 순으로 상위 3개 표시 |

> **TTS를 1개로 제한하는 이유**: 러닝 중 사용자는 동시에 2개 이상의 음성 정보를 처리하기 어렵다. 음성이 겹치면 정보 전달이 모두 실패한다. 따라서 음성은 가장 행동 가치가 높은 1개 메시지에 집중한다. 화면은 동시 인지가 가능하므로 3개까지 허용한다.
> 

> **Android의 TTS queue 관리 책임**: 본 모듈이 cycle당 1개로 제한하더라도, Android는 자체적으로 다음을 관리해야 한다.
> 
> - 현재 TTS가 재생 중인지 확인
> - 재생 중이면 새 메시지를 큐에 넣을지 / 무시할지 결정
> - 분석 종료 시 큐에 남은 메시지 처리 (즉시 종료 또는 마지막까지 재생)
> 
> 본 모듈은 *"이 메시지는 TTS로 출력하라"* 신호만 보내며, TTS 재생 시점 / 큐 / 인터럽트는 Android의 책임이다.
> 

### 3-6. 시스템 안내와 자세 경고 동시 발생 시 처리

시스템 안내(reason code 활성화)와 자세 경고가 동시 발생하는 경우, 다음 정책으로 처리한다.

| 상황 | 본 모듈 동작 |
| --- | --- |
| `failed` reason code 활성화 + 자세 피드백 활성화 | **자세 피드백 생성 안 함**. reason code 메시지만 출력 |
| `low_confidence` reason code 활성화 + 자세 피드백 활성화 | reason code 메시지를 우선순위 1로 음성 출력. 자세 피드백은 신뢰도 prefix와 함께 화면 중심으로 표시 |
| reason code 없음 + 자세 피드백만 활성화 | 자세 피드백 정상 출력 |

이 정책은 3-4절(status별 피드백 정책) + 3-5절(출력 한도)의 조합이며, 명시적 정리이다.

## 4. 트리거 룰 (분석 결과 → 피드백 메시지)

본 장은 본 시스템의 모든 피드백 트리거 룰의 **유일한 정의 위치**이다. 본 표의 모든 메시지는 **초기 default 문구**이며, 사용자 테스트를 통해 보정한다.

각 메시지는 음성용 짧은 문구(`tts_text`)와 화면용 자세한 문구(`display_text`)로 분리되어 정의된다. `tts_text`는 러닝 중 듣기 쉽도록 짧게(약 1초 분량), `display_text`는 정보량을 더 담는다.

### 4-1. Foot Strike Pattern 관련

| 트리거 조건 | tts_text | display_text | 카테고리 | TTS 기본 |
| --- | --- | --- | --- | --- |
| 누적 분류 = `RFS` | (TTS 출력 없음) | "현재 뒤꿈치 중심 착지(rearfoot strike) 패턴이 관찰됩니다." | 자세 안내 | **False** |
| 누적 분류 = `MFS` | (TTS 출력 없음) | "현재 발 중간 부위 착지(midfoot strike) 패턴이 관찰됩니다." | 자세 안내 | **False** |
| 누적 분류 = `FFS` | (TTS 출력 없음) | "현재 앞발 중심 착지(forefoot strike) 패턴이 관찰됩니다." | 자세 안내 | **False** |
| 누적 분류가 `Uncertain`로 5 stride 이상 지속 | "착지 패턴이 안정적이지 않아요." | "착지 패턴 측정이 안정적이지 않습니다. 자세를 일정하게 유지해주세요." | 시스템 안내 | True |

> **Foot Strike Pattern을 기본 TTS False로 두는 이유**: Foot Strike Pattern은 *"분류 자체에 우월/열등이 없는 정보성 지표"* 다(2-3-4 5-5절). RFS/MFS/FFS가 매번 음성으로 나와도 사용자가 즉시 행동을 바꾸기 어렵고, 오히려 피로감을 유발한다. 따라서 다음 정책을 기본으로 한다.
> 
> - 실시간: 화면 표시만
> - 분석 종료 시: 최종 요약에서 제공
> - 예외: `Uncertain` 지속은 시스템 안내로 TTS 출력 (행동 수정 가능성 있음)

> **가치 판단 회피 원칙 적용**: Foot Strike Pattern은 *"당신의 착지 패턴은 X입니다"* 정도의 통계적 서술이며, *"X로 바꾸세요"* 같은 지시는 포함하지 않는다.
> 

### 4-2. Initial Knee Flexion 관련

| 트리거 조건 | tts_text | display_text | 카테고리 | TTS 기본 |
| --- | --- | --- | --- | --- |
| 누적 분류 = `Below Typical Range` | "무릎 굴곡이 작은 편이에요." | "무릎 굴곡이 일반 범위(15~25°)보다 작게 관찰됩니다. 착지 충격을 줄이기 위해 무릎을 조금 더 부드럽게 사용해보세요." | 자세 경고 | True |
| 누적 분류 = `Typical Range` | (TTS 출력 없음) | "착지 시 무릎 굴곡이 일반 범위에 있습니다." | 자세 안내 (빈도 제한 강화) | False |
| 누적 분류 = `Above Typical Range` | "무릎이 많이 굽혀지고 있어요." | "착지 시 무릎 굴곡이 일반 범위보다 큽니다. 충격 흡수에는 유리할 수 있지만 에너지 효율이 낮아질 수 있습니다." | 자세 경고 | True |

> **`Below Typical Range`의 메시지 톤 보수화**: 이전 *"무릎을 자연스럽게 굽혀보세요"* 같은 직접적 행동 지시는 가치 판단 회피 원칙과 긴장 관계가 있어, *"착지 충격을 줄이기 위해 …해보세요"* 형태의 **참고용 제안** 톤으로 약화했다. 충격 흡수가 부족할 가능성이라는 정보를 제공하되, 진단/지시가 아닌 참고 정보임을 유지한다.
> 

> **`Above Typical Range`는 trade-off 명시**: Above Typical Range는 충격 흡수에 유리한 측면과 에너지 효율 저하 측면의 trade-off가 있어, 일방적 행동 지시 없이 trade-off만 통계적으로 서술한다.
> 

### 4-3. Trunk Lean 관련

| 트리거 조건 | tts_text | display_text | 카테고리 | TTS 기본 |
| --- | --- | --- | --- | --- |
| 누적 분류 = `Near Vertical` | (TTS 출력 없음) | "상체가 거의 수직에 가깝습니다. 약간의 전방 기울기는 러닝 효율에 도움이 될 수 있습니다." | 자세 안내 | False |
| 누적 분류 = `Forward Lean` | (TTS 출력 없음) | "상체 전경사가 일반 범위에 있습니다." | 자세 안내 (빈도 제한 강화) | False |
| 누적 분류 = `Above Typical Range` | "상체가 많이 기울어져 있어요." | "상체가 일반 범위보다 더 앞으로 기울어져 있습니다. 자세가 무너지지 않도록 주의해보세요." | 자세 경고 | True |

### 4-4. 측정 분산 관련 (지표별 공통)

각 지표의 측정 분산이 큰 경우의 트리거.

| 트리거 조건 | tts_text | display_text | 카테고리 | TTS 기본 |
| --- | --- | --- | --- | --- |
| 지표의 사분위 범위(IQR) > 분류 임계 폭의 50% | "자세를 일정하게 유지해주세요." | "측정 분산이 큰 편입니다. 자세를 일정하게 유지해주세요." | 시스템 안내 | True |

> **분산이 큰 경우의 메시지 의미**: 측정값 자체는 어느 분류에 속해 있더라도, stride마다 값이 크게 흔들린다면 그 분류 자체가 의미가 약해진다. 사용자에게 *"자세 안정화"* 를 요청하는 메시지가 더 의미 있다.
> 

### 4-5. 시스템 상태 관련 (긍정 메시지)

분석이 안정적으로 진행되고 있을 때의 긍정 피드백.

| 트리거 조건 | tts_text | display_text | 카테고리 | TTS 기본 |
| --- | --- | --- | --- | --- |
| 누적 분석 시작 직후 (첫 stride 3개 누적) | "분석을 시작합니다." | "분석을 시작합니다." | 시스템 안내 | True |
| 모든 지표가 Typical Range + 분산 작음 (30초 이상 간격) | "좋은 페이스 유지해주세요." | "현재 자세가 안정적입니다. 좋은 페이스 유지해주세요." | 자세 안내 (긴 빈도 제한) | **True (예외)** |
| 분석 종료 시점 | "분석이 완료되었습니다." | "분석이 완료되었습니다. 결과를 확인해주세요." | 시스템 안내 | True |

> **긍정 메시지의 TTS 예외**: 일반 `posture_info`는 기본 TTS False이지만, *"좋은 페이스 유지해주세요"* 같은 안정적 자세 유지 메시지는 30초 이상 긴 빈도 제한 하에서 동기 부여 효과가 있어 TTS True로 출력 가능하다. 빈도가 충분히 낮으면 fatigue가 발생하지 않는다.
>

## 5. 시스템 안내 (Reason code 인용)

분석 품질 문제로 인한 시스템 안내 메시지는 본 문서가 직접 생성하지 않고, **2-3-5 문서 8장의 reason code 사전을 인용**한다.

| 단일 정답 위치 원칙 | 적용 |
| --- | --- |
| 분석 품질 관련 메시지 (재촬영 안내 등) | 2-3-5 문서 8장 인용 (`too_short`, `camera_unstable`, `body_not_fully_visible` 등) |
| 자세 분석 결과 관련 메시지 | 본 문서 4장이 정의 |

> **두 메시지 종류의 분리 이유**: Reason code 메시지는 *"분석 자체가 어려운 상황"* (재촬영 필요)을 다루며, 본 문서의 트리거 룰은 *"분석은 정상적으로 진행되고 있고, 자세에 대한 피드백을 제공"* 한다. 두 종류의 메시지는 사용자 입장에서도 다른 행동을 요구하므로 출처를 분리하는 것이 자연스럽다.
>

## 6. 피드백 메시지 출력 구조

본 모듈은 응답 메시지(2-3-7)에 포함될 피드백 텍스트를 다음 구조로 후속 단계에 전달한다.

```
{
    "feedback_messages": [
        {
            "category": "posture_warning",         # posture_warning, posture_info, system_info
            "metric": "initial_knee_flexion",      # 어떤 지표 관련인가 (선택)
            "tts_text": "무릎 굴곡이 작은 편이에요.",
            "display_text": "무릎 굴곡이 일반 범위(15~25°)보다 작게 관찰됩니다. 착지 충격을 줄이기 위해 무릎을 조금 더 부드럽게 사용해보세요.",
            "priority": 2,
            "tts_enabled": True,                   # Android에서 음성 합성할지 여부
            "confidence_prefix": False             # low_confidence 상태에서 신뢰도 안내 prefix 동반
        }
    ]
}
```

### 6-1. tts_text와 display_text의 분리

| 필드 | 용도 | 길이 가이드 |
| --- | --- | --- |
| `tts_text` | TTS 음성 합성용 짧은 문구 | 약 1초 분량 (한국어 기준 10~15자) |
| `display_text` | 화면 표시용 자세한 문구 | 30~60자 정도, 정보량 우선 |

> **두 필드의 분리 이유**: 러닝 중 음성으로 듣는 정보는 짧고 즉각적이어야 한다. 한 메시지에 *"착지 시 무릎 굴곡이 일반 범위보다 큽니다. 충격 흡수에 유리하지만 에너지 효율이 낮을 수 있습니다"* 가 들어가면 TTS 재생 시간이 5초 이상 걸려 다음 메시지를 막는다. 음성은 핵심만 짧게, 화면은 trade-off 정보까지 담아 사용자가 분석 종료 후 천천히 읽을 수 있도록 분리한다.
> 

> **MVP 구현 가이드**: 본 v1에서 두 필드 분리가 부담된다면, `tts_text`만 사용하고 `display_text`는 동일 텍스트로 채워도 무방하다. 다만 출력 구조는 이미 분리되어 있어, 이후 사용자 테스트 후 두 필드를 다르게 채우기만 하면 된다.
> 

### 6-2. 카테고리 분류

| 카테고리 | 의미 | 우선순위 | TTS 기본 |
| --- | --- | --- | --- |
| `system_info` | 시스템 안내 (재촬영, 분석 시작/종료, 측정 분산 등) | 1 (최우선) | True |
| `posture_warning` | 자세 경고 (Above/Below Typical Range) | 2 | True |
| `posture_info` | 자세 안내 (Typical Range 정보 / Foot Strike 분류 등) | 3 | False (예외 4-5절) |

### 6-3. `tts_enabled` 필드의 역할

피드백 메시지가 화면에만 표시될지, 음성으로도 출력될지 본 모듈이 결정한다.

| 카테고리 | `tts_enabled` 기본값 | 비고 |
| --- | --- | --- |
| `system_info` | True (음성 출력) |  |
| `posture_warning` | True (음성 출력) |  |
| `posture_info` (Foot Strike Pattern 등) | **False** (화면에만) | 정보성 지표는 음성 반복 시 피로 |
| `posture_info` (긍정 안정 메시지) | True (예외) | 30초 이상 긴 빈도 제한 하에 출력 |

### 6-4. `confidence_prefix` 필드의 역할

`low_confidence` 상태에서 자세 피드백을 출력할 때 신뢰도 안내 prefix를 함께 표시할지 결정한다.

| status | confidence_prefix | 결과 |
| --- | --- | --- |
| `success` | False | 메시지 그대로 출력 |
| `low_confidence` | True | "측정 신뢰도가 낮아 참고용으로만 확인해주세요." prefix 동반 |
| `failed` | (해당 없음) | 자세 피드백 자체가 생성되지 않음 |

## 7. 다른 모듈/파트와의 인터페이스

### 7-1. AI 파트 내부 인터페이스

| 본 문서가 정의 | 사용/연동 문서 | 연동 방식 |
| --- | --- | --- |
| 누적 통계 입력 | 2-3-4 (자세 지표 산출) | stride 누적 결과를 입력으로 받음 |
| status 입력 | 2-3-5 (분석 상태값) | success/low_confidence/failed에 따라 출력 정책 분기 |
| Reason code 인용 | 2-3-5 (8장 reason code 사전) | `system_info` 메시지는 2-3-5의 사용자 메시지를 그대로 사용 |
| 피드백 메시지 출력 | 2-3-7 (응답 메시지 구조화) | `feedback_messages` 필드로 응답 메시지에 포함 |
| 가치 판단 회피 원칙 | 2-3-4 (분류 라벨 정책) | 본 문서 4장 메시지가 2-3-4의 typical range 정책과 정합 |

### 7-2. Android 인터페이스

| 본 문서가 정의 | Android 처리 방식 |
| --- | --- |
| 피드백 메시지 텍스트 (`tts_text`, `display_text`) | 화면에 `display_text` 표시 (`category`별 UI 분기), 음성으로는 `tts_text` 합성 |
| `tts_enabled` 플래그 | True인 경우만 TTS 엔진으로 음성 합성 |
| 메시지 우선순위 | 한 응답에 여러 메시지가 있으면 우선순위대로 음성 출력 (Android TTS queue 관리) |
| 출력 한도 (TTS 1개, 화면 3개) | 본 모듈이 이미 한도를 적용하여 응답하므로, Android는 받은 메시지를 그대로 출력 |
| 빈도 제한 | 본 모듈이 이미 빈도 제한을 적용하여 응답하므로, Android는 받은 메시지를 그대로 출력 |
| TTS queue 관리 (재생 중 인터럽트, 큐잉, 종료 시 정리) | Android 자체 책임 |

> **Android의 책임 경계**: Android는 *"무엇을 말할지"* 를 결정하지 않는다. 그건 본 모듈이 결정한다. Android는 *"어떻게 말할지"* (음성 합성, UI 표시 스타일, TTS queue 관리)만 책임진다. 이 분리로, 피드백 정책 변경 시 Android 코드는 수정할 필요가 없다.
>

## 8. 메시지 보정 계획

본 문서의 트리거 룰과 메시지 문구는 사용자 테스트를 거치며 다음 항목을 보정한다.

| 보정 항목 | 보정 시점 | 검증 방법 |
| --- | --- | --- |
| `tts_text` 문구 | 사용자 테스트 단계 | 사용자가 1초 안에 메시지를 들었는지, 행동 의도를 파악했는지 확인 |
| `display_text` 문구 | 사용자 테스트 단계 | 사용자가 메시지를 읽고 trade-off 정보를 이해하는지 확인 |
| 빈도 제한 (5초/2초/30초) | 시연 단계 | 피드백 fatigue 발생 여부 관찰 |
| 우선순위 | 시연 단계 | 우선순위 충돌 케이스에서 적절한 메시지가 출력되는지 |
| **status별 출력 정책 적합성** | 시연 단계 | low_confidence에서 신뢰도 prefix가 사용자 신뢰에 미치는 영향 확인 |
| **출력 한도 (TTS 1개, 화면 3개)** | 시연 단계 | 동시 트리거 케이스에서 사용자 인지 부담 확인 |
| 분산 임계 (IQR > 50%) | 파일럿 단계 | 분산이 큰 영상에서 트리거되는 빈도 확인 |
| `tts_enabled` 정책 (특히 Foot Strike False) | 사용자 테스트 단계 | 어떤 카테고리가 음성으로 들리는 게 적절한지 |
| 첫 피드백 출력 시점 | 시연 단계 | 실측 지연이 2~3초 목표 범위에 들어오는지 |

## 9. 최종 정리

본 문서는 AI 파트의 Rule-based 실시간 피드백 정책의 단일 정답 위치이다.

핵심 결정 사항을 다시 정리하면 다음과 같다.

1. 본 모듈은 **누적 통계 기반**으로만 피드백을 생성하며, 단일 stride 즉시 피드백은 사용하지 않는다.
2. **첫 피드백 출력은 분석 시작 후 약 2~3초 목표** (3 stride 누적 + IC 검출/누적/처리 지연 포함 실측 범위).
3. **`success` / `low_confidence` / `failed` 상태별 피드백 정책이 다르다.** `failed`는 자세 피드백 차단, `low_confidence`는 신뢰도 prefix 동반.
4. **TTS는 cycle당 최대 1개, 화면은 최대 3개**까지 출력한다. 음성 겹침으로 인한 정보 전달 실패를 방지.
5. 메시지는 **`tts_text`(음성용 짧은 문구)와 `display_text`(화면용 자세한 문구)로 분리**된다.
6. 피드백 메시지는 **가치 판단을 부여하지 않는다.** Typical range 외/내라는 통계적 서술을 사용한다.
7. **동일 메시지는 5초 내 중복 출력 안 함**, 서로 다른 메시지 간 **2초 최소 간격** 적용.
8. 우선순위는 **시스템 안내 > 자세 경고 > 자세 안내** 순서.
9. **Foot Strike Pattern은 기본 TTS False** (정보성 지표라 음성 반복 시 피로). `Uncertain` 지속 시에만 TTS 출력.
10. **본 모듈은 텍스트 메시지를 출력**하며, 실제 음성 합성과 TTS queue 관리는 Android의 책임이다.
11. 시스템 품질 안내 메시지(재촬영 등)는 **2-3-5의 reason code 사전을 인용**하며, 본 문서가 직접 정의하지 않는다.

피드백 메시지 문구와 빈도 제한 값은 모두 초기 default이며, 사용자 테스트를 통해 보정한다.
