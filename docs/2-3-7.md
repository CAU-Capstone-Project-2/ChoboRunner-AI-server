2-3-7. 자세 분석 결과 구조화

## 1. 이 문서를 한눈에

### 이 문서가 정의하는 것

이 문서는 **"AI 서버가 Spring을 통해 Android에 응답하는 모든 메시지의 스펙"** 을 정의합니다. AI 파트 안에서 이 문서는 다음 세 가지의 **단일 정답 위치(single source-of-truth)** 역할을 합니다.

- **메시지 종류와 구조** (frame별 즉시 응답, 진행 상태, 최종 누적 응답, 에러)
- **모든 필드의 이름, 타입, 단위** (백엔드/Android에서 그대로 사용)
- **메시지별 발생 시점과 트리거 조건**

### 이 문서가 정의하지 않는 것

다음 항목은 다른 문서에 정의되어 있으니 그쪽을 참조해주세요.

| 알고 싶은 것 | 가야 할 문서 |
| --- | --- |
| WebSocket 연결 설정, 세션 관리, 재연결 전략은? | 백엔드 WebSocket 설계 문서 |
| Reason code의 의미와 사용자 메시지 문구는? | **2-3-5** (8장 reason code 사전) |
| Reason code 우선순위 (사용자 노출 1개 선택 규칙)는? | **2-3-5** (8-7절) |
| 분석 상태값(success / low_confidence / failed)의 결정 기준은? | **2-3-5** (6장 분기 룰) |
| 메타데이터 임계값(영상 길이, fps 등)은? | **2-3-1** (3장 임계값 표) |
| 자세 지표(Foot Strike, Knee Flexion, Trunk Lean) 계산식은? | **2-3-4** (자세 지표 산출) |
| 피드백 메시지 트리거 룰과 메시지 사전은? | **2-3-6** (Rule-based 실시간 피드백) |
| Frame-level 품질 검사(휘도, 블러, 흔들림)는? | **2-3-2** (4장 품질 검사 3종) |

### 다른 파트가 알아두면 좋은 핵심 결정 6가지

이 문서에서 결정된 사항 중, 다른 파트(백엔드, 프론트엔드)에 영향을 주는 핵심 정책은 다음과 같습니다.

1. **응답 메시지는 4종류뿐.** Frame별 즉시 응답(`frame_inference`), 진행 상태(`analysis_progress`), 최종 누적 응답(`analysis_result`), 에러(`error`). 다른 메시지는 정의되지 않으며, Android는 이 4종류만 처리하면 된다.
2. **모든 자세 지표는 단일값으로 응답된다.** `initial_knee_flexion_deg`, `trunk_lean_deg`, `foot_strike_pattern` — 좌우 분리 응답 없음. 분석측은 `analysis_side: "left" | "right"`로 명시된다.
3. **각도 단위는 모두 도(degree)이며, 모든 각도는 정해진 reference axis 기준이다.** Trunk Lean은 수직축 기준, Foot angle은 수평축 기준. 정상 러너의 trunk_lean_deg는 5~10° 범위.
4. **사용자 노출 reason code는 `primary_reason_code` 1개로 좁힌다.** 내부 평가 reason code는 `reason_codes` 배열에 모두 포함된다. 우선순위 결정 규칙은 2-3-5 8-7절.
5. **피드백 메시지는 `feedback_messages` 배열로 응답된다.** 2-3-6의 트리거 룰로 생성된 메시지가 `analysis_result` 또는 `analysis_progress`의 optional 필드로 포함된다.
6. **Frame별 즉시 응답은 디버그/모니터링 성격이다.** 사용자 화면에 직접 표시하지 않는다. 사용자에게 보여줄 결과는 `analysis_result` 메시지에서만 받는다.

## 2. 메시지 종류 개요

본 시스템의 응답 메시지는 4종류이며, 모두 `type` 필드로 구분된다.

| 메시지 종류 | `type` 값 | 발생 시점 | 사용처 |
| --- | --- | --- | --- |
| Frame별 즉시 응답 | `frame_inference` | 매 frame 추론 직후 | 디버그, 모니터링, 실시간 시연 |
| 진행 상태 | `analysis_progress` | 누적 분석 단계 변경 시 또는 stride 갱신 시 | 사용자 화면의 "분석 중" 상태 + **실시간 피드백** |
| 최종 누적 응답 | `analysis_result` | 분석 종료 시 (사용자 종료 또는 frame 수신 중단) | **사용자 결과 화면의 단일 정답 위치** |
| 에러 | `error` | 시스템 처리 실패 시 (메시지 파싱 실패, 내부 예외 등) | Android 에러 처리 |

### 메시지 흐름 예시

```
[Android] ────── frame 1 ───────▶ [AI 서버]
                                       │
[Android] ◀── frame_inference ────────┤    (frame 1 추론 즉시 응답)
[Android] ────── frame 2 ───────▶     │
[Android] ◀── frame_inference ────────┤
                                       │
[Android] ◀── analysis_progress ──────┤    ("분석 준비 중", warming_up)
                                       │
[Android] ────── frame 50 ──────▶     │
[Android] ◀── frame_inference ────────┤
                                       │
[Android] ◀── analysis_progress ──────┤    ("분석 중", collecting_strides)
                                       │
[Android] ────── frame 100 ─────▶     │
[Android] ◀── frame_inference ────────┤
                                       │
[Android] ◀── analysis_progress ──────┤    ("결과 갱신", analyzing
                                       │     + feedback_messages 포함)
                                       │
[Android] ──── 분석 종료 신호 ──────▶  │
[Android] ◀── analysis_result ────────┤    (최종 누적 결과 + feedback_messages)
```

## 3. Frame별 즉시 응답 (`frame_inference`)

### 3-1. 발생 시점과 용도

매 frame이 AI 서버에 도착하여 추론이 완료된 직후 즉시 응답된다. 본 메시지는 **디버그, 모니터링, 실시간 시연용 성격**이며, 사용자 화면에 직접 표시하지 않는다.

> **이 메시지를 사용자 화면에 표시하지 않는 이유**: 단일 frame의 추론 결과는 노이즈에 매우 민감하다. 분석 대상 frame의 측정값이 stride마다 ±5° 이상 변동하는 것이 정상이며(2-3-4 부록 D 참조), 이 값을 그대로 사용자에게 보여주면 *"왜 값이 계속 바뀌지?"* 라는 혼란을 유발한다. 사용자 결과는 stride 누적 후의 `analysis_result`에서만 확인한다.
> 

### 3-2. 메시지 구조

```json
{
  "type": "frame_inference",
  "frame_index": 42,
  "timestamp_sec": 1.400,
  "result": {
    "pose_detected": true,
    "frame_quality_flags": []
  }
}
```

### 3-3. 필드 정의

| 필드 | 타입 | 단위 | 설명 |
| --- | --- | --- | --- |
| `type` | string | — | 항상 `"frame_inference"` |
| `frame_index` | int | — | 도착 순서 frame 번호 (0부터) |
| `timestamp_sec` | float | sec | capture timestamp 기준 경과 시간 (2-3-2 3-3절) |
| `result.pose_detected` | bool | — | MediaPipe Pose 추출 성공 여부 |
| `result.frame_quality_flags` | string[] | — | 2-3-2의 품질 플래그 배열 (비어있으면 품질 통과) |

`frame_quality_flags`에 들어갈 수 있는 값: `low_brightness`, `motion_blur`, `frame_unstable`, `timestamp_fallback` (정의는 2-3-2 4장).

### 3-4. Frame error vs System error 경계

본 문서에서는 다음 정책으로 두 차원을 구분한다.

| 상황 | 메시지 표현 |
| --- | --- |
| 단일 frame의 Pose 추출 실패 (사람을 못 찾음) | `frame_inference.result.pose_detected = false` |
| 단일 frame의 디코딩 실패 또는 품질 미달 | `frame_inference.result.frame_quality_flags`에 플래그 추가 |
| 시스템 처리 실패 (내부 예외, 메시지 파싱 실패 등) | `type: "error"` 별도 메시지 |

> **`frame_inference`는 항상 `status: ok`로 응답된다.** Frame 단위의 모든 비정상 상황은 `result` 내부 필드(`pose_detected: false`, `frame_quality_flags`)로 표현되며, 별도 status 필드를 두지 않는다. 진짜 시스템 에러는 `type: "error"` 메시지로 분리된다. 이로써 *"frame error인지 system error인지"* 의 모호함이 해소된다.
> 

### 3-5. 메시지에서 제외된 필드와 그 이유

백엔드 임시 응답에는 다음 필드들이 있었으나, 본 정의에서는 제외했다.

| 제외된 필드 | 제외 이유 |
| --- | --- |
| `width`, `height` | Frame별 응답에 포함시키지 않음. 영상 메타데이터는 `analysis_result`의 `video_meta`에 포함 |
| `visible_landmarks` | 단일 frame의 visibility 카운트는 분석에 활용되지 않음. 누적 visibility는 `analysis_result`에 포함 |
| `metrics` (좌우 분리된 무릎 각도 등) | 단일 frame의 자세 지표값은 노이즈 변동이 크므로 사용자 화면에 노출하지 않음. 최종 누적값은 `analysis_result`에 포함 |
| `metadata` (빈 객체) | 사용처가 정해지지 않은 빈 필드는 메시지 정의에서 제외 |
| `status` 필드 | 3-4절 정책에 따라 `frame_inference`는 항상 ok이므로 status 필드 자체를 제거 |

> **시연 시 디버그 표시 권고**: 시연 자리에서 *"실시간으로 분석되는 모습"* 을 보여주고 싶다면, 별도 디버그 화면에서 `frame_inference`의 `frame_index`와 `pose_detected`만 표시하는 것이 자연스럽다. 자세 지표값을 frame별로 표시하는 것은 권장되지 않는다.
>

## 4. 진행 상태 메시지 (`analysis_progress`)

### 4-1. 발생 시점과 용도

다음 두 경우에 응답된다.

1. **누적 분석의 단계가 변경될 때** (stage 전환)
2. **`analyzing` 단계에서 stride가 갱신되어 실시간 피드백을 보낼 때**

사용자 화면의 *"분석 중"* 상태 표시 + **실시간 피드백 메시지 전달**에 사용된다.

### 4-2. 메시지 구조 (단계 전환)

```json
{
  "type": "analysis_progress",
  "stage": "collecting_strides",
  "valid_stride_count": 2,
  "elapsed_sec": 3.5,
  "message": "분석 중입니다."
}
```

### 4-3. 메시지 구조 (실시간 피드백 포함)

`analyzing` 단계에서 stride가 추가될 때마다 누적 통계가 갱신되며, 2-3-6의 트리거 룰에 의해 피드백 메시지가 생성된다. 이 경우 `feedback_messages` 필드가 함께 포함된다.

```json
{
  "type": "analysis_progress",
  "stage": "analyzing",
  "valid_stride_count": 4,
  "elapsed_sec": 4.2,
  "message": "분석 중입니다.",
  "feedback_messages": [
    {
      "category": "posture_warning",
      "metric": "trunk_lean",
      "tts_text": "상체가 많이 기울어져 있어요.",
      "display_text": "상체가 일반 범위보다 더 앞으로 기울어져 있습니다. 자세가 무너지지 않도록 주의해보세요.",
      "priority": 2,
      "tts_enabled": true,
      "confidence_prefix": false
    }
  ]
}
```

### 4-4. 필드 정의

| 필드 | 타입 | 단위 | 응답 조건 | 설명 |
| --- | --- | --- | --- | --- |
| `type` | string | — | 항상 | 항상 `"analysis_progress"` |
| `stage` | string | — | 항상 | 분석 단계 (4-5절 참조) |
| `valid_stride_count` | int | — | 항상 | 누적된 유효 stride 수 |
| `elapsed_sec` | float | sec | 항상 | 분석 시작 후 경과 시간 |
| `message` | string | — | 선택 | 사용자 표시용 진행 상태 문구 |
| `feedback_messages` | object[] | — | 선택 | 2-3-6 트리거 룰로 생성된 실시간 피드백 메시지 배열 |

### 4-5. 분석 단계 (`stage` 값)

| `stage` 값 | 의미 | 사용자 화면 권장 표시 | feedback_messages 포함 가능 |
| --- | --- | --- | --- |
| `warming_up` | 첫 IC 검출 전 (frame 누적 중) | "분석 준비 중..." | No |
| `collecting_strides` | 첫 IC 검출 완료, 유효 stride 3개 미만 | "분석 중..." | No |
| `analyzing` | 유효 stride 3개 누적 후, 매 stride마다 결과 갱신 가능 | "분석 중... (결과 갱신)" | **Yes** |

`analyzing` 단계 도중에 사용자가 분석 종료 신호를 보내면 `analysis_result`로 전환된다.

### 4-6. `feedback_messages` 필드 구조

본 필드의 구조는 4종 메시지 모두에서 동일하다. 자세한 정의는 5-7절 참조.

> **2-3-6과의 정합**: 본 필드의 메시지는 2-3-6 문서의 트리거 룰에 따라 생성된다. 2-3-6은 *"무슨 말을 할지"* 를 결정하고, 본 문서는 *"어떤 JSON 구조로 전달할지"* 를 정의한다.
>

## 5. 최종 누적 응답 (`analysis_result`)

### 5-1. 발생 시점과 용도

분석 종료 시점(사용자가 분석 종료를 요청하거나, 일정 시간 frame 수신이 멈춘 시점)에 1회 응답된다. **사용자 결과 화면의 단일 정답 위치이며, 프론트엔드는 이 메시지의 데이터를 그대로 결과 화면에 표시한다.**

### 5-2. 메시지 구조 (success)

```json
{
  "type": "analysis_result",
  "status": "success",
  "video_meta": {
    "duration_sec": 12.4,
    "fps_actual": 29.8,
    "resolution": {
      "width": 1280,
      "height": 720
    },
    "total_frames": 369
  },
  "analysis_side": "right",
  "metrics": {
    "foot_strike_pattern": "MFS",
    "foot_strike_angle_deg": 2.3,
    "initial_knee_flexion_deg": 19.4,
    "trunk_lean_deg": 7.8
  },
  "metric_details": {
    "foot_strike_angle_deg": {
      "median": 2.3,
      "iqr": [1.0, 3.4],
      "n_strides": 5
    },
    "initial_knee_flexion_deg": {
      "median": 19.4,
      "iqr": [17.8, 21.2],
      "n_strides": 5
    },
    "trunk_lean_deg": {
      "median": 7.8,
      "iqr": [6.5, 9.1],
      "n_strides": 5
    }
  },
  "quality_summary": {
    "valid_frame_ratio": 0.84,
    "ic_candidate_count": 6,
    "valid_stride_count": 5,
    "landmark_visibility_avg": 0.78,
    "target_tracking_stability": "stable"
  },
  "primary_reason_code": null,
  "reason_codes": [],
  "message": "분석이 완료되었습니다.",
  "feedback_messages": [
    {
      "category": "posture_info",
      "metric": "foot_strike_pattern",
      "tts_text": null,
      "display_text": "현재 발 중간 부위 착지(midfoot strike) 패턴이 관찰됩니다.",
      "priority": 3,
      "tts_enabled": false,
      "confidence_prefix": false
    }
  ]
}
```

> **`success`에서 `message` 필드는 선택**: *"분석이 완료되었습니다"* 같은 결과 요약 한 줄을 담을 수 있다. 자세한 자세 피드백은 `feedback_messages`에 위임되며, `message`는 화면 상단의 짧은 요약 문구 용도이다. 응답에 포함시키지 않아도 무방하다.
> 

### 5-3. 메시지 구조 (low_confidence)

```json
{
  "type": "analysis_result",
  "status": "low_confidence",
  "video_meta": { ... },
  "analysis_side": "right",
  "metrics": {
    "foot_strike_pattern": "MFS",
    "foot_strike_angle_deg": 1.8,
    "initial_knee_flexion_deg": 17.2,
    "trunk_lean_deg": 8.4
  },
  "metric_details": { ... },
  "quality_summary": { ... },
  "primary_reason_code": "low_ic_confidence",
  "reason_codes": ["low_ic_confidence", "unstable_foot_angle"],
  "message": "착지 시점을 안정적으로 찾기 어렵습니다. 다시 촬영해주세요.",
  "feedback_messages": [
    {
      "category": "posture_warning",
      "metric": "initial_knee_flexion",
      "tts_text": "무릎 굴곡이 작은 편이에요.",
      "display_text": "무릎 굴곡이 일반 범위(15~25°)보다 작게 관찰됩니다. 착지 충격을 줄이기 위해 무릎을 조금 더 부드럽게 사용해보세요.",
      "priority": 2,
      "tts_enabled": true,
      "confidence_prefix": true
    }
  ]
}
```

> **`low_confidence` 정책**: 핵심 지표 3개가 모두 계산 가능하므로 `metrics`와 `metric_details`를 정상적으로 응답한다. 다만 신뢰도가 낮은 상태이므로 `message`와 `reason_codes`를 함께 응답하여 재촬영을 권장한다. `feedback_messages`의 자세 피드백 항목은 `confidence_prefix: true`로 응답되어, Android가 *"측정 신뢰도가 낮아 참고용으로만 확인해주세요"* 안내 prefix를 자동으로 붙인다 (2-3-6 v2 3-4절과 정합).
> 
> 
> 결과 화면은 결과를 표시하면서 동시에 신뢰도 안내를 함께 표시하며, 재촬영 버튼을 노출한다.
> 

### 5-4. 메시지 구조 (failed)

```json
{
  "type": "analysis_result",
  "status": "failed",
  "video_meta": {
    "duration_sec": 2.3,
    "fps_actual": 30.0,
    "resolution": { "width": 1280, "height": 720 },
    "total_frames": 69
  },
  "primary_reason_code": "too_short",
  "reason_codes": ["too_short"],
  "message": "러닝 구간이 너무 짧아 분석이 어렵습니다. 최소 10초 이상 달려주세요."
}
```

> **`failed`는 자세 피드백 차단**: `failed` 케이스는 `metrics`, `metric_details`, `analysis_side`, `quality_summary`, `feedback_messages` 필드가 모두 응답되지 않는다. 분석 자체가 실패했으므로 자세 피드백을 보내는 것은 부적절하다 (2-3-6 v2 3-4절과 정합). 사용자에게는 `message`(reason code 사용자 메시지)만 표시된다.
> 

### 5-5. 필드 정의

### 최상위 필드

| 필드 | 타입 | 응답 조건 | 설명 |
| --- | --- | --- | --- |
| `type` | string | 항상 | `"analysis_result"` |
| `status` | string | 항상 | `"success"`, `"low_confidence"`, `"failed"` 중 하나 |
| `video_meta` | object | 항상 | 영상 메타데이터 누적 결과 |
| `analysis_side` | string | success / low_confidence | `"left"` 또는 `"right"` |
| `metrics` | object | success / low_confidence | 핵심 지표 3개의 대표값 |
| `metric_details` | object | success / low_confidence | 지표별 통계 (median, IQR, n_strides) |
| `quality_summary` | object | success / low_confidence | 품질 검사 요약 |
| `primary_reason_code` | string | null | 항상 | 사용자 노출용 대표 reason code (1개), success는 null |
| `reason_codes` | string[] | 항상 | 트리거된 모든 reason code 배열 (success는 빈 배열) |
| `message` | string | success(선택) / low_confidence / failed | 사용자에게 표시할 메시지 (`primary_reason_code` 기준) |
| `feedback_messages` | object[] | success / low_confidence | 2-3-6의 트리거 룰로 생성된 피드백 메시지 배열 |

### `primary_reason_code`와 `reason_codes`의 관계

| 필드 | 의미 | 사용처 |
| --- | --- | --- |
| `primary_reason_code` | 사용자 노출용 대표 reason code (1개) | 결과 화면 메시지 표시 |
| `reason_codes` | 내부 평가에서 트리거된 모든 reason code | 디버그, 로그 분석 |

> **선택 규칙**: 여러 reason code가 동시에 활성화될 때, `primary_reason_code`는 **2-3-5 8-7절의 reason code 우선순위 체계**에 따라 1개로 좁혀진다. `message` 필드도 `primary_reason_code`에 해당하는 사용자 메시지(2-3-5 8장)를 사용한다.
> 
> 
> **예시**: `reason_codes = ["foot_out_of_frame", "low_landmark_visibility", "lower_body_not_visible"]` → 8-7절 우선순위에 따라 `primary_reason_code = "foot_out_of_frame"` 선택 → `message = "발이 화면 아래에서 잘립니다. 카메라를 조금 멀리 두세요."`
> 

### `video_meta` 필드

| 필드 | 타입 | 단위 | 설명 |
| --- | --- | --- | --- |
| `duration_sec` | float | sec | 누적 분석 시간 |
| `fps_actual` | float | frames/sec | 실측 평균 fps (2-3-2의 effective FPS와 동일 의미) |
| `resolution` | object | — | `{ width, height }` (정규화 후 해상도) |
| `total_frames` | int | — | 누적 frame 수 |

> **`resolution`의 의미**: 본 필드는 2-3-2의 cap 정규화(상한선 정규화)를 거친 후의 해상도이다. 입력 영상이 1080p였다면 `width: 1280, height: 720`로 응답된다. 입력 해상도와 처리 해상도를 분리해서 응답하는 것은 향후 확장 영역이다 (디버그 / `low_resolution` 원인 분석 등에 유용하지만, MVP에서는 단순화 우선).
> 

### `metrics` 필드

| 필드 | 타입 | 단위 | 설명 | 분류 라벨 |
| --- | --- | --- | --- | --- |
| `foot_strike_pattern` | string | — | 착지 유형 분류 결과 | `"RFS"`, `"MFS"`, `"FFS"` |
| `foot_strike_angle_deg` | float | degree | Foot angle (수평축 기준, 양수 = dorsiflexion) | — |
| `initial_knee_flexion_deg` | float | degree | 무릎 굴곡 각도 (180° - 관절각) | — |
| `trunk_lean_deg` | float | degree | 상체 전경사 각도 (수직축 기준, 양수 = 전방 기울기) | — |

> **각도 reference axis 정의**:
> 
> - `foot_strike_angle_deg`: 영상 수평축 기준. heel-to-toe 벡터의 각도. 양수 = 발끝이 들린 상태.
> - `trunk_lean_deg`: 영상 수직축 기준. shoulder-hip 벡터의 각도. 정상 러너는 5~10° 범위.
> - `initial_knee_flexion_deg`: 180° - hip-knee-ankle 관절각. 직선에 가까울수록 0°에 가까움.

### `metric_details` 필드

각 지표별 stride 단위 통계.

| 필드 | 타입 | 단위 | 설명 |
| --- | --- | --- | --- |
| `median` | float | degree | 중앙값 (= `metrics`의 값과 동일) |
| `iqr` | float[2] | degree | 사분위 범위 `[Q1, Q3]` |
| `n_strides` | int | — | 통계 계산에 사용된 stride 수 |

> **median을 사용하는 이유와 IQR 표시 이유**: 단일 stride outlier에 대한 강건성 때문에 평균이 아닌 중앙값을 쓴다(2-3-4 8-2절). IQR은 측정 분산을 보여주어 사용자가 *"이 측정이 얼마나 안정적이었나"* 를 직관적으로 파악할 수 있다.
> 

### `quality_summary` 필드

| 필드 | 타입 | 단위 | 설명 |
| --- | --- | --- | --- |
| `valid_frame_ratio` | float | — | 최종 분석 후보 frame 비율 (0.0 ~ 1.0) |
| `ic_candidate_count` | int | — | 검출된 전체 IC 후보 수 |
| `valid_stride_count` | int | — | 통계 계산에 사용된 유효 stride 수 |
| `landmark_visibility_avg` | float | — | 주요 landmark visibility 평균 (0.0 ~ 1.0) |
| `target_tracking_stability` | string | — | `"stable"`, `"borderline"`, `"unstable"` 중 하나 |

> **`valid_frame_ratio` 정의**: 다음 3가지를 **모두 만족**해 최종 분석 후보로 사용 가능한 frame의 비율이다.
> 
> 1. Pose detection 성공 (`pose_detected = true`, 2-3-3 4-2절)
> 2. Frame-level 품질 통과 (2-3-2 플래그가 비어있음 — `low_brightness`, `motion_blur`, `frame_unstable` 모두 없음)
> 3. Landmark visibility 임계 통과 (2-3-5 5-1절 — 분석측 hip/knee/ankle visibility 평균 0.6 이상)
> 
> 단순 `pose_detected` 비율이나 visibility 통과 비율과 다르다. 본 필드는 *"실제로 분석에 사용 가능한 frame이 몇 %였는가"* 를 의미한다.
> 

### `foot_strike_pattern` 분류 라벨

| 라벨 | 의미 | 영문 |
| --- | --- | --- |
| `"RFS"` | 뒤꿈치 중심 착지 | Rearfoot Strike |
| `"MFS"` | 중족부 착지 | Midfoot Strike |
| `"FFS"` | 앞발 중심 착지 | Forefoot Strike |

> 본 시스템은 분류 결과에 가치 판단을 부여하지 않는다(2-3-4 5-5절). 라벨은 통계적 분류이며, 어느 것이 우월하다는 의미가 아니다.
> 

### 5-6. `feedback_messages` 배열 정의

본 필드는 2-3-6 문서의 트리거 룰에 따라 생성된 피드백 메시지 배열이다. 본 문서는 메시지의 JSON 구조만 정의하며, **메시지 내용 / 트리거 조건 / 출력 정책은 모두 2-3-6이 단일 정답 위치이다.**

### 배열 요소 구조

```json
{
  "category": "posture_warning",
  "metric": "initial_knee_flexion",
  "tts_text": "무릎 굴곡이 작은 편이에요.",
  "display_text": "무릎 굴곡이 일반 범위(15~25°)보다 작게 관찰됩니다. 착지 충격을 줄이기 위해 무릎을 조금 더 부드럽게 사용해보세요.",
  "priority": 2,
  "tts_enabled": true,
  "confidence_prefix": false
}
```

### 필드 정의

| 필드 | 타입 | 설명 |
| --- | --- | --- |
| `category` | string | `"system_info"`, `"posture_warning"`, `"posture_info"` 중 하나 (2-3-6 6-2절) |
| `metric` | string | null | 어떤 지표 관련 메시지인가. 시스템 안내는 null |
| `tts_text` | string | null | TTS 음성 합성용 짧은 문구 (1초 분량). TTS 출력 안 할 메시지는 null 가능 |
| `display_text` | string | 화면 표시용 자세한 문구 |
| `priority` | int | 1(최우선), 2, 3 (2-3-6 3-3절 우선순위) |
| `tts_enabled` | bool | Android에서 TTS 합성할지 여부 |
| `confidence_prefix` | bool | true면 Android는 *"측정 신뢰도가 낮아 참고용으로만 확인해주세요"* prefix를 붙인다 (low_confidence 케이스) |

### 출력 한도 (2-3-6 3-5절 정책 인용)

본 배열에는 cycle당 최대 **3개**까지 메시지가 포함된다. 그중 `tts_enabled: true`인 메시지는 **1개**만 포함된다 (음성 겹침 방지).

> **2-3-6의 정책 적용 위치**: 출력 한도(TTS 1개, 화면 3개), 빈도 제한(동일 메시지 5초, 다른 메시지 2초), 우선순위 등은 모두 2-3-6이 결정한다. 본 문서가 받는 시점에는 이미 정책이 적용된 후의 메시지이며, Android는 받은 메시지를 그대로 출력하면 된다.
>

## 6. 에러 메시지 (`error`)

### 6-1. 발생 시점과 용도

AI 서버 내부에서 시스템 처리 실패가 발생했을 때 응답된다. 분석 결과의 성공/실패와는 다른 차원의 시스템 에러를 다룬다.

> **`failed` 상태와 `error` 메시지의 차이**: `analysis_result`의 `status: "failed"` 는 *"입력 영상이 분석 조건을 만족하지 못함"* (정상 처리 후의 비즈니스 결과)이다. `error` 메시지는 *"AI 서버 자체에 처리 실패가 발생함"* (시스템 에러)이다.
> 

### 6-2. 메시지 구조

```json
{
  "type": "error",
  "frame_index": 42,
  "error_code": "internal_processing_failed",
  "error_detail": "MediaPipe 추론 중 예외 발생"
}
```

### 6-3. 필드 정의

| 필드 | 타입 | 단위 | 설명 |
| --- | --- | --- | --- |
| `type` | string | — | 항상 `"error"` |
| `frame_index` | int | null | — | 에러가 발생한 frame 번호 (해당 없으면 null) |
| `error_code` | string | — | 에러 코드 (구현 단계에서 확정) |
| `error_detail` | string | — | 디버그용 상세 메시지 |

> **`error_code` 목록은 구현 단계에서 확정**: 현재 단계에서는 메시지 구조만 정의하고, 구체적인 에러 코드는 AI 서버 구현 시 발견되는 실제 에러 케이스에 맞춰 보강한다. 예시: `internal_processing_failed`, `mediapipe_exception`, `out_of_memory` 등.
>

## 7. 다른 모듈/파트와의 인터페이스

### 7-1. AI 파트 내부 인터페이스

| 본 문서가 정의 | 사용하는 문서 | 사용 방식 |
| --- | --- | --- |
| 응답 메시지 4종 구조 | (없음) | 본 문서가 단일 정답 위치 |
| `analysis_side` 필드 | 2-3-4, 2-3-5 | 분석측 단일값 정책의 출력 채널 |
| `primary_reason_code` 필드 | 2-3-5 (8-7절 우선순위 체계) | 사용자 노출 reason code 1개 선택 |
| `reason_codes` 필드 | 2-3-5 (8장 reason code 사전) | 내부 평가 reason code 목록 |
| `metric_details` 필드 (median/IQR) | 2-3-4 (자세 지표 산출) | 8-2절 누적 통계 결과 |
| `quality_summary` 필드 | 2-3-5 (품질 검사) | 누적 품질 검사 결과 |
| `feedback_messages` 필드 | 2-3-6 (Rule-based 피드백) | 2-3-6이 생성한 메시지를 본 구조로 직렬화 |

### 7-2. 백엔드 인터페이스

| 본 문서가 정의 | 백엔드 처리 방식 |
| --- | --- |
| 메시지 구조 (4종) | Spring은 AI 서버가 응답한 JSON을 그대로 Android에 relay (구조 변경하지 않음) |
| `type` 필드 | Spring은 `type` 필드를 보지 않고 단순 relay |
| 메시지 크기 | 모든 메시지가 1KB 미만으로 설계됨 (binary frame과 달리 메시지 크기 제한과 무관) |

> **백엔드의 단순 릴레이 원칙**: 본 문서의 메시지 구조는 AI 서버가 생성하고, Spring은 변환 없이 Android로 전달한다. Spring이 메시지를 파싱/변환하지 않는 이유는, 메시지 구조 변경 시 두 곳을 수정해야 하는 비용을 피하기 위함이다.
> 

### 7-3. 프론트엔드 인터페이스

| 본 문서가 정의 | Android 처리 방식 |
| --- | --- |
| `type: "frame_inference"` | 디버그 로그 또는 별도 디버그 화면에만 표시 (사용자 결과 화면에 표시 안 함) |
| `type: "analysis_progress"` (단계 전환) | "분석 중..." 상태 화면에서 단계별 메시지 표시 |
| `type: "analysis_progress"` (feedback_messages 포함) | 상태 화면 표시 + 실시간 피드백 출력 (TTS / 화면) |
| `type: "analysis_result"` (success) | 결과 화면에 `metrics` + `analysis_side` + `feedback_messages` 표시 |
| `type: "analysis_result"` (low_confidence) | 결과 + `message` 표시 + 재촬영 버튼 + `feedback_messages` (신뢰도 prefix 포함) |
| `type: "analysis_result"` (failed) | 재촬영 안내 화면에 `message` 표시 (자세 피드백 없음) |
| `type: "error"` | 시스템 에러 다이얼로그 또는 토스트 |

### `feedback_messages` Android 처리 가이드

| 필드 | Android 동작 |
| --- | --- |
| `display_text` | 화면 카드 / 토스트로 표시 (`category`별 UI 분기) |
| `tts_text` | `tts_enabled: true`인 경우 TTS 엔진으로 음성 합성 |
| `tts_enabled` | true인 메시지만 TTS 합성. 한 응답에 1개만 true임이 보장됨 |
| `confidence_prefix` | true인 경우 *"측정 신뢰도가 낮아 참고용으로만 확인해주세요"* 안내를 메시지 앞에 표시 |
| `priority` | 여러 메시지의 표시 순서 결정 (1이 가장 위) |

### 결과 화면 표시 권고

`analysis_result` (`success` / `low_confidence`) 메시지를 받았을 때 결과 화면에 표시할 정보:

| 영역 | 표시할 내용 | 출처 필드 |
| --- | --- | --- |
| 분석측 표시 | "오른쪽 다리 기준" | `analysis_side` |
| 핵심 지표 3개 | 라벨 + 값 + 단위 | `metrics` |
| 측정 신뢰도 표시 | "측정 5회 기준 (17.8 ~ 21.2 범위)" | `metric_details.iqr`, `n_strides` |
| 분석 시간 | "12.4초 분석" | `video_meta.duration_sec` |
| 자세 피드백 | 카드 형식으로 메시지 표시 | `feedback_messages[].display_text` |
| 저신뢰 케이스 | 재촬영 권유 + 신뢰도 prefix | `message`, `feedback_messages[].confidence_prefix` |

## 8. 메시지 스펙 보정 계획

본 문서의 메시지 구조는 구현/시연을 거치며 다음 항목을 보정한다.

| 보정 항목 | 보정 시점 | 검증 방법 |
| --- | --- | --- |
| `error_code` 구체 목록 | AI 서버 구현 단계 | 발생하는 실제 에러 케이스 수집 후 분류 |
| `analysis_progress`의 `stage` 값 | 시연 단계 | 사용자 UX 관점에서 "분석 중" 단계가 어색하지 않은지 확인 |
| `metric_details`의 추가 필드 | 시연 후 사용자 피드백 | 사용자가 *"이 측정이 얼마나 안정적인가"* 를 직관적으로 파악하는지 |
| `quality_summary`의 추가 필드 | 시연 단계 | 디버그/문제 분석에 부족한 정보가 없는지 |
| `feedback_messages`의 한 응답 내 빈도 | 시연 단계 | analysis_progress에 메시지가 포함되는 빈도가 적절한지 |
| `primary_reason_code` 선택 결과 | 시연 단계 | 다양한 실패 케이스에서 사용자에게 표시되는 메시지가 적절한지 |
| `video_meta.resolution` input/processed 분리 여부 | 향후 확장 | `low_resolution` 디버그 시 input 해상도가 필요한지 |

## 9. 최종 정리

본 문서는 AI 파트의 응답 메시지 스펙의 단일 정답 위치이다.

핵심 결정 사항을 다시 정리하면 다음과 같다.

1. 응답 메시지는 **4종**: `frame_inference`(디버그), `analysis_progress`(진행 상태 + 실시간 피드백), `analysis_result`(최종 결과), `error`(시스템 에러).
2. 사용자 결과 화면은 **`analysis_result`만** 사용한다. `frame_inference`는 디버그/시연용이며 사용자에게 직접 노출하지 않는다.
3. 모든 자세 지표는 **단일값으로 응답**된다 (`analysis_side` 필드로 분석측 명시).
4. **사용자 노출 reason code는 `primary_reason_code` 1개**, 내부 평가는 `reason_codes` 배열 전체. 우선순위 결정 규칙은 2-3-5 8-7절.
5. **`feedback_messages` 배열로 피드백 메시지 전달**. `analysis_progress`(실시간)와 `analysis_result`(최종) 모두에서 응답 가능. 메시지 구조는 `tts_text` / `display_text` / `confidence_prefix` 분리 (2-3-6 v2와 정합).
6. **`low_confidence`는 결과 제공 + 재촬영 권장**. `metrics`와 `metric_details`를 정상 응답하되 `message`와 `confidence_prefix: true`로 신뢰도 안내를 동반.
7. **`failed`는 자세 피드백 차단**. `metrics`, `metric_details`, `feedback_messages` 모두 응답하지 않음.
8. 각도 단위는 모두 **degree**, 각도의 reference axis는 본 문서 5-5절에 정의되어 있다.
9. **`error` 메시지는 `failed`와 다른 차원**: `failed`는 비즈니스 결과(분석 조건 미달), `error`는 시스템 에러. `frame_inference`는 항상 `status: ok`이며, frame 단위 비정상은 `result` 내부 필드로 표현.
10. Spring 백엔드는 본 문서의 메시지를 **변환 없이 단순 relay**한다.

WebSocket 통신 인프라(연결 수립, 세션 관리 등)에 대한 결정은 백엔드 WebSocket 설계 문서를 참조한다. Reason code의 구체 정의와 사용자 메시지는 2-3-5 문서를 참조한다. 피드백 메시지의 트리거 룰과 메시지 사전은 2-3-6 문서를 참조한다.
