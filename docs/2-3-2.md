2-3-2. 영상 전처리 및 프레임 품질 검사

## 1. 이 문서를 한눈에

### 이 문서가 정의하는 것

이 문서는 **"MediaPipe Pose 추출 직전에 frame을 정리하고 frame 자체의 품질을 검사하는 단계"** 의 정책을 정의합니다. AI 파트 안에서 이 문서는 다음 두 가지의 **단일 정답 위치(single source-of-truth)** 역할을 합니다.

- **영상 정규화 정책** (fps, 해상도, frame timestamp 관리)
- **Frame-level 품질 검사 기준** (평균 휘도, Laplacian variance, frame-level 흔들림)

### 이 문서가 정의하지 않는 것

다음 항목은 다른 문서에 정의되어 있으니 그쪽을 참조해주세요.

| 알고 싶은 것 | 가야 할 문서 |
| --- | --- |
| 영상 길이/FPS/해상도 같은 메타데이터 임계값은? | **2-3-1** (3장 임계값 표) |
| 전신 포함 / 발 잘림 / 측면 구도 검사는? | **2-3-5** (5장 — Pose 추출 후 검사) |
| Landmark visibility / 분석 대상자 추적 안정성은? | **2-3-5** |
| MediaPipe Pose API 호출 / 모델 선택은? | **2-3-3** |
| `low_brightness`, `motion_blur` 같은 reason code의 사용자 메시지는? | **2-3-5** (8장 reason code 사전) |
| WebSocket 메시지 형식, frame binary 전송 방식은? | 백엔드 WebSocket 설계 문서 |

### 다른 파트가 알아두면 좋은 핵심 결정 4가지

이 문서에서 결정된 사항 중, 다른 파트(백엔드, 프론트엔드, 다른 AI 모듈)에 영향을 주는 핵심 정책은 다음과 같습니다.

1. **영상 정규화는 상한선 정규화(cap) 방식이다.** 30fps와 720p를 목표 상한선으로 사용한다. 초과 입력은 30fps / 720p로 downsample하고, 미만 입력은 인위적으로 보간/업스케일하지 않고 원본을 유지한다.
2. **시간축은 Android capture timestamp를 기준으로 사용한다.** Android는 frame capture 시점의 system timestamp를 함께 전송하고, AI 서버는 이를 IC 검출 등 모든 시간 계산의 기준으로 사용한다. AI 서버 수신 시각은 latency 모니터링 용도로만 기록된다.
3. **품질 미달 frame을 즉시 거부하지 않는다.** 품질 플래그를 부여하고 후속 단계(2-3-4 / 2-3-5)가 stride 누적에서 가중치 0으로 처리한다. 단일 frame의 일시적 품질 미달이 즉시 분석 실패로 이어지지 않는다.
4. **Frame-level 품질 검사는 픽셀 수준 신호만 본다.** 평균 휘도, Laplacian variance, 인접 frame 간 픽셀 변화량 3종. Landmark 기반 검사(전신 포함, 측면 구도 등)는 본 단계가 아닌 2-3-5에서 한다.

## 2. 본 단계의 위치

본 단계는 WebSocket으로 수신한 binary frame을 받아 정규화하고, **MediaPipe Pose 추출 직전**에 frame-level 품질 검사를 수행한다.

```
[Android]
   │  binary frame (JPEG 압축) + capture timestamp
   ▼
[Spring]   ─── 단순 릴레이
   │
   ▼
[AI 서버]
   │
   ├─ [2-3-1 메타데이터 검증] ─ 즉시 검증 (해상도, decode 가능 여부)
   │
   ▼
[2-3-2 영상 전처리 + Frame-level 품질 검사 — 본 문서]
   │  ├ JPEG 디코딩
   │  ├ fps / 해상도 정규화 (cap)
   │  ├ Timestamp 정규화 (capture timestamp 기준)
   │  └ Frame-level 품질 검사 3종 → 품질 플래그
   │
   ▼
[2-3-3 MediaPipe Pose 추출]
   │
   ▼
[2-3-4 자세 지표 산출]
   │
   ▼
[2-3-5 측정 품질 판단]
   │
   ▼
[2-3-6 피드백 생성]
   │
   ▼
[2-3-7 결과 메시지 응답]
```

### 두 층으로 나누는 이유

본 단계와 2-3-5의 분리 원칙:

- **본 단계 (2-3-2)**: **픽셀 수준 신호만** 본다. MediaPipe 호출 없이 빠르게 검사 가능. frame-level 단위로 비싼 Pose 추론을 건너뛰는 1차 필터 역할.
- **Pose 추출 후 검사 (2-3-5)**: **landmark 좌표 기반 신호**(visibility, 신체 위치 관계)를 본다. MediaPipe 결과가 있어야 검사 가능.

이 분리는 *"자기 자신을 보지 못하는 검사를 같은 단계에서 시도하지 않는다"* 는 원칙에 따른다. 예를 들어 *"전신이 화면에 포함되었는가"* 를 Pose 추출 전 단계에서는 알 수 없다 (landmark 좌표 없이 사람이 어디 있는지 모름).

## 3. 영상 정규화

본 장은 WebSocket으로 수신한 binary frame을 후속 단계가 사용할 수 있는 형태로 정규화하는 정책을 정의한다.

### 3-1. fps 정규화 (Cap 정규화)

### 정규화 정책

본 시스템은 **30fps를 목표 상한선**으로 사용한다.

| 입력 fps | 처리 |
| --- | --- |
| 30fps 초과 (예: 60fps, 50fps, 45fps) | 30fps timestamp grid에 가장 가까운 frame만 채택 |
| 정확히 30fps | 그대로 유지 |
| 30fps 미만 (예: 24fps, 20fps) | 그대로 유지. 인위적 보간으로 30fps로 끌어올리지 않음 |

### Downsampling 일반 원칙

30fps 초과 입력은 **목표 timestamp grid(33.3ms 간격)에 가장 가까운 frame**을 채택한다. 60fps 입력은 본 원칙의 특수 사례로 매 2 frame 중 1 frame이 채택되며, 50fps, 45fps 같은 비균등 fps도 동일 원칙으로 처리된다.

### 30fps 미만 입력을 보간하지 않는 이유

인위적 보간 frame은 IC 검출에서 false signal을 만들 수 있다. 24fps 영상을 30fps로 보간하면 보간된 frame의 발 좌표가 실제 촬영된 좌표와 다르며, 이 false 좌표가 IC 검출의 zero-crossing에 영향을 준다. 따라서 미만 입력은 원본 fps를 그대로 유지하고, 누적 분석 시간 / 누적 frame 수는 2-3-1 메타데이터 임계 검증으로 처리된다.

### 2-3-4와의 시간축 정합

본 단계가 timestamp 기반으로 시간축을 관리하므로, 후속 모듈(특히 2-3-4의 IC 검출)이 사용하는 frame 단위 파라미터(예: `±3 frame` 윈도우, `15 frame` 최소 IC 간격)는 **effective FPS 기준으로 ms 단위 시간으로 해석**되어야 한다.

| 파라미터 | 30fps 입력 | 24fps 입력 |
| --- | --- | --- |
| `±3 frame` 윈도우 | 100ms | 125ms |
| `15 frame` 최소 IC 간격 | 500ms | 625ms |
| `LOOKAHEAD = 3` | 100ms | 125ms |

향후 2-3-4 보강 시 frame 기반 파라미터를 ms 기반으로 병기할 예정이다. 본 문서가 timestamp를 모든 시간 계산의 기준으로 두는 것은, 이러한 fps별 환산 부담을 후속 모듈이 지지 않도록 하기 위함이다.

### 3-2. 해상도 정규화 (Cap 정규화)

본 시스템은 **720p(긴 변 1280px)를 목표 상한선**으로 사용한다.

| 입력 해상도 | 처리 |
| --- | --- |
| 720p 초과 (예: 1080p, 4K) | 긴 변 1280px로 downsample (OpenCV `cv2.resize`, INTER_AREA 보간) |
| 정확히 720p | 그대로 유지 |
| 720p 미만 (예: 480p) | 그대로 유지. 인위적 업스케일하지 않음 |

720p 미만 입력은 2-3-1 메타데이터 검증에서 `low_resolution` reason code로 `failed` 처리된다 (2-3-1 문서 3장 임계값 표 참조).

### 해상도 cap 720p의 근거

- MediaPipe Pose가 내부적으로 입력 해상도를 변환하므로 1080p 대비 정확도 차이가 크지 않다.
- 모바일 업로드 및 WebSocket 전송 비용 측면에서 720p가 균형점이다.
- 720p로 downsample하면 후속 처리(JPEG 디코딩, Pose 추론)의 메모리/연산 비용이 1080p 대비 약 56% 감소한다.

### 3-3. Frame timestamp 관리

WebSocket 실시간 스트림은 frame drop, 네트워크 지연, 비균등 frame 간격이 발생할 수 있으므로, 본 단계는 **timestamp 기반 시간축**으로 frame을 관리한다. Frame index는 도착 순서일 뿐 시간을 의미하지 않는다.

### Timestamp 기준: Android capture timestamp

**MVP 정책**: Android는 카메라 frame capture 시점의 system timestamp(`System.nanoTime()` 또는 `SystemClock.elapsedRealtimeNanos()`)를 함께 전송하고, AI 서버는 이 capture timestamp를 기준 시간축으로 사용한다. AI 서버 수신 시각은 latency 모니터링 용도로만 기록하며, IC 검출 / stride time 계산 등 시간 정확도가 중요한 처리에는 사용하지 않는다.

### Capture timestamp를 기준으로 사용하는 이유

| timestamp 후보 | 정확도 | 의미 |
| --- | --- | --- |
| **Android capture timestamp** (선택) | 0~5ms 오차 | 실제 촬영 시점에 가까움 |
| Android 송신 시각 | 5~50ms 오차 | JPEG 인코딩 후 전송 시각 |
| Spring 릴레이 시각 | 추가 지연 | 서버 간 hop 시간 포함 |
| AI 서버 수신 시각 | 50~200ms 오차 | 네트워크 전송 지연 포함 |

IC 검출은 ms 단위 정확도가 필요하므로(2-3-4 문서 4장 참조), 가능한 가장 정확한 시간 기준을 사용해야 한다. capture timestamp는 카메라 콜백에서 받는 OS 수준 timestamp이므로 가장 정확하다.

### Fallback 정책

Capture timestamp가 누락된 frame이 도착한 경우(Android 구현 누락, 메시지 포맷 변경 등), AI 서버 수신 시각을 fallback으로 사용한다. 이 경우 해당 frame에 `timestamp_fallback` 플래그를 부여하여 시간 정확도가 낮음을 표시한다. 분석 도중 fallback 사용 비율이 일정 이상(예: 30% 이상) 누적되면 2-3-5에 신호를 전달한다.

### 백엔드/Android 구현 영향

본 정책은 백엔드 / Android 구현 가이드에 영향을 준다.

- **Android**: WebSocket binary message에 capture timestamp를 포함시켜야 한다. 메시지 포맷(예: header에 8 byte timestamp 추가, 또는 별도 metadata frame 전송)은 백엔드 WebSocket 설계 문서에서 정의한다.
- **Spring**: capture timestamp를 그대로 AI 서버에 릴레이한다. 시간 변환 / 보정은 하지 않는다.

### Frame drop 처리

Frame drop이 발생해도 본 단계는 frame을 거부하지 않는다. 도착한 frame을 그대로 처리하며, timestamp 기반으로 시간 계산이 이루어지므로 누락된 frame이 있어도 후속 단계의 시간 정확도에 영향이 없다.

### 3-4. 디코딩 정책

WebSocket으로 수신한 binary frame은 JPEG 압축된 형태로 도착한다(2-3-1 문서 2장 참조). 본 단계는 다음 순서로 디코딩한다.

1. JPEG binary → image array (OpenCV `cv2.imdecode`)
2. 해상도 cap 정규화 (3-2절)
3. Frame timestamp 메타데이터 부여 (3-3절)
4. Frame-level 품질 검사 (4장)

디코딩 실패한 frame은 `decode_failed` 플래그를 부여하고, 2-3-1의 즉시 검증 단계(첫 frame 검증)와 별개로 본 단계에서는 후속 단계로 전달하지 않는다(품질 플래그 정책의 예외).

## 4. Frame-level 품질 검사 3종

본 단계는 MediaPipe Pose 추출 비용을 절감하고 사전 필터링을 위해 다음 3종의 frame-level 품질 검사를 수행한다. 본 장의 모든 임계값은 **초기 default 값**이며, 파일럿 데이터 검증을 통해 보정한다.

### 4-1. 평균 휘도 검사 (조도 부족 검출)

| 항목 | 내용 |
| --- | --- |
| 검사 신호 | Frame을 grayscale로 변환한 후 픽셀 평균 |
| 임계값 (default) | 평균 휘도 < 50 (0~255 스케일 기준) → 조도 부족 |
| 처리 | Frame에 `low_brightness` 품질 플래그 부여, frame은 그대로 후속 단계로 전달 |
| 미충족 시 reason code | `low_brightness` (2-3-5 8장) |

> **임계값 50의 근거**: 일반적인 실내 헬스장 조명에서 grayscale 평균은 80~150 범위에 있다. 50 미만은 매우 어두운 환경(조명 꺼짐, 야간 등)으로 MediaPipe Pose의 visibility가 광범위하게 떨어지는 임계 영역이다.
> 

### 4-2. 모션 블러 검사

| 항목 | 내용 |
| --- | --- |
| 검사 신호 | Laplacian variance (이미지의 2차 미분 분산) |
| 임계값 (default) | Laplacian variance < 100 → 모션 블러 |
| 처리 | Frame에 `motion_blur` 품질 플래그 부여, frame은 그대로 후속 단계로 전달 |
| 미충족 시 reason code | `motion_blur` (2-3-5 8장) |

> **Laplacian variance의 원리**: 이미지의 엣지가 선명할수록 Laplacian 분산이 높다. 모션 블러로 엣지가 뭉개지면 분산이 급격히 감소한다. OpenCV의 `cv2.Laplacian()` + `np.var()`로 빠르게 계산 가능 (frame당 약 5ms).
> 
> 
> **임계값 100의 근거**: 일반적인 정상 frame은 Laplacian variance 200~1000 범위에 있다. 100 미만은 사람이 빠르게 움직이는 frame에서 발생하며, MediaPipe의 landmark 위치 정확도가 떨어지는 임계 영역이다.
> 

### 4-3. Frame-level 흔들림 검사

| 항목 | 내용 |
| --- | --- |
| 검사 신호 | 인접 frame 간 픽셀 변화량 (간단 버전: SSD; 정밀 버전: optical flow magnitude 평균) |
| 임계값 (default) | 픽셀 변화량 > 평균의 200% → 비정상적 변동 |
| 처리 | Frame에 `frame_unstable` 품질 플래그 부여, frame은 그대로 후속 단계로 전달 |
| 미충족 시 reason code | `frame_unstable` (2-3-5 8장) |

> **Frame-level 흔들림과 Stride-level 흔들림의 구분**: Frame-level은 **인접 frame 간 픽셀 변화**(빠른 카메라 흔들림)를 본다. Stride-level은 **stride 단위 pelvis 좌표 변동**(촬영 자세의 일관성)을 본다. 두 신호는 다른 종류의 흔들림을 잡는다. Stride-level 흔들림 검사는 2-3-5가 담당한다.
> 
> 
> **구현 권고**: 초기 구현은 SSD(Sum of Squared Differences) 기반 간단 버전으로 시작하고, 정확도가 부족하면 optical flow magnitude로 업그레이드한다. SSD는 frame당 1~2ms, optical flow는 10~20ms 수준.
> 

### SSD의 한계 인지

SSD는 인접 frame 간 전체 픽셀 차이를 보므로, **러닝 자체의 빠른 신체 움직임을 카메라 흔들림으로 오탐할 수 있다.** 러닝 영상은 본질적으로 사람의 팔/다리가 빠르게 움직이는 영상이며, 정상 러닝 동작도 SSD 기준에서 큰 변화량을 보일 수 있다.

이러한 한계로 인해 본 검사의 의미는 *"카메라 흔들림"* 의 단정 판정이 아니라 **frame-level 이상 변동 신호** 정도로 해석한다. 최종 *"카메라가 흔들렸는가"* 판정은 stride 단위 pelvis 좌표 변동을 보는 2-3-5가 한다.

정확도가 필요한 경우 다음 보강 기법을 향후 도입 가능하다 (향후 확장 영역).

- **Background ROI 기반**: frame 가장자리(사람이 보통 없는 영역)만 비교하여 카메라 자체 움직임 추정
- **Optical flow 저주파 분리**: 전체 픽셀 이동의 저주파(전역 이동 = 카메라 panning)와 고주파(국소 이동 = 신체 움직임) 분리
- **Pelvis 영역 제외 ROI**: 사람이 가장 큰 영역에 위치하는 영상 중앙 일부를 제외하고 비교

### 4-4. 품질 플래그 처리 원칙

본 단계의 품질 검사는 **frame을 거부하지 않는다.** 모든 frame은 품질 플래그와 함께 후속 단계로 전달된다.

| 처리 단계 | 정책 |
| --- | --- |
| 본 단계 (2-3-2) | 품질 플래그 부여, frame은 그대로 전달 |
| 2-3-3 (Pose 추출) | 모든 frame에 대해 Pose 추출 수행 (품질 플래그 무관) |
| 2-3-4 (지표 산출) | 품질 플래그가 있는 frame은 stride 누적에서 가중치 0으로 처리 (제외) |
| 2-3-5 (최종 상태 결정) | 품질 플래그 비율이 일정 이상이면 reason code 활성화 |

> **이 정책의 이유**: 단일 frame의 품질 미달이 즉시 분석 실패로 이어지지 않도록 하기 위함이다. 일시적인 모션 블러나 카메라 흔들림은 stride 누적 과정에서 자연스럽게 흡수될 수 있다. 최종 거부 결정은 누적 데이터를 본 2-3-5가 한다.
> 

> **향후 확장 영역**: 다중 품질 플래그(`low_brightness` + `motion_blur` + `frame_unstable` 동시 활성화)가 발생한 frame은 사실상 분석 가치가 없으므로 Pose 추출을 생략하는 2단계 정책도 가능하다. MVP에서는 단순성 우선으로 모든 frame에 Pose 추출을 수행하지만, 후속 모듈의 처리 비용이 병목이 될 경우 본 절에 추출 생략 정책을 추가한다.
>

## 5. 본 단계의 처리 비용

WebSocket 실시간 스트림 모드에서는 frame당 처리 시간이 30fps 한 frame 예산(33ms) 안에 들어와야 한다. 본 단계의 추정 처리 비용은 다음과 같다.

| 단계 | 추정 시간 (per frame) |
| --- | --- |
| JPEG 디코딩 | ~5ms |
| 해상도 정규화 (1080p → 720p) | ~3ms |
| 평균 휘도 검사 | ~1ms |
| Laplacian variance 검사 | ~5ms |
| Frame-level 흔들림 검사 (SSD 버전) | ~1ms |
| **본 단계 합계** | **~15ms** |

본 추정치는 설계 단계의 예상치이며, 구현 단계에서 실측 검증한다.

> **실시간 처리 가정의 보수적 명시**: 본 단계 ~15ms는 30fps 한 frame 예산(33ms)의 절반 미만이다. 다만 후속 MediaPipe Pose 추출(~33ms)과 단순 합산하면 frame 예산을 초과하므로, **실시간 처리는 비동기 파이프라인 또는 frame skip 전략을 전제로 한다.** 본 단계와 Pose 추출의 통합 latency는 구현 단계에서 실측 검증 대상이며, 다음 옵션 중 하나로 대응한다.
> 
> - **비동기 파이프라인**: 본 단계와 Pose 추출을 별도 thread/process로 분리하여 병렬 처리
> - **Frame skip**: 30fps → 15fps로 frame을 일부 건너뛰어 후속 단계 부하 감소
> - **MediaPipe `lite` 모델**: Pose 추출 비용을 절반 이하로 단축 (정확도 trade-off)

## 6. 본 단계의 출력 구조

본 단계는 frame을 처리한 후 다음 구조로 후속 단계(2-3-3)에 전달한다.

```python
{
    "frame_index": 0,                     # 도착 순서 (시간 의미 없음)
    "timestamp_sec": 0.000,               # capture timestamp 기준 (3-3절)
    "image": <decoded image array>,       # 720p 정규화 frame
    "quality_flags": [],                  # 품질 검사 결과 (빈 배열 = 품질 통과)
    "fps_actual_recent": 30.0,            # sliding window 기반 실측 fps
}
```

`quality_flags`에 들어갈 수 있는 값:

| 플래그 | 의미 |
| --- | --- |
| `low_brightness` | 평균 휘도 임계 미달 |
| `motion_blur` | Laplacian variance 임계 미달 |
| `frame_unstable` | Frame-level 흔들림 임계 초과 |
| `timestamp_fallback` | capture timestamp 누락으로 AI 서버 수신 시각 사용 |

여러 플래그가 동시에 활성화될 수 있다 (예: `["motion_blur", "frame_unstable"]`).

### `fps_actual_recent` 정의

본 필드는 **sliding window 기반 실측 fps**이다.

- **계산 방식**: 최근 30 frame의 인접 timestamp 간격 평균의 역수 (`1 / mean(intervals_ms) × 1000`)
- **갱신 시점**: frame이 도착할 때마다 갱신
- **의미**: 현재 frame 시점의 **최근 1초 평균 fps**
- **다른 fps 개념과의 차이**:
    - **nominal fps** (Android 카메라 설정값): 본 필드와 다르다. nominal 30fps여도 인코딩/네트워크 지연으로 actual은 더 낮을 수 있다.
    - **누적 평균 fps**: 분석 시작부터 현재까지의 전체 평균. 본 필드는 sliding window 기반이라 최근 변동을 반영한다.
    - **2-3-1의 effective FPS**: 분석 종료 시점의 누적 평균. 본 필드는 매 frame 갱신되는 실시간 값이라는 점이 다르다.

> **응답 JSON 메시지와의 차이**: 본 출력은 **AI 서버 내부의 모듈 간 데이터 구조**이며, Spring/Android로 응답되는 메시지 구조가 아니다. 응답 메시지는 2-3-7에서 별도 정의된다.
>

## 7. 다른 모듈/파트와의 인터페이스

### 7-1. AI 파트 내부 인터페이스

| 본 문서가 정의 | 사용하는 문서 | 사용 방식 |
| --- | --- | --- |
| 30fps × 720p cap 정규화 frame 출력 | 2-3-3 (Pose 추출) | 정규화된 frame을 입력으로 받음 |
| Capture timestamp 기반 frame timestamp | 2-3-4 (IC 검출) | 모든 시간 계산은 timestamp 기준 (frame 단위 파라미터는 effective FPS로 ms 환산) |
| 품질 플래그 (`low_brightness`, `motion_blur`, `frame_unstable`, `timestamp_fallback`) | 2-3-4 (stride 누적), 2-3-5 (최종 상태) | 플래그 frame은 stride에서 제외, 비율 누적은 reason code 트리거 |
| 누적 메타데이터 (`fps_actual_recent`) | 2-3-1 (메타데이터 임계 검증) | 분석 종료 시점 effective FPS 계산의 입력 |

### 7-2. 백엔드 인터페이스

| 본 문서가 정의 | 백엔드 처리 방식 | 비고 |
| --- | --- | --- |
| Frame 수신 | Spring이 binary frame을 AI 서버에 그대로 릴레이 | 메시지 크기 제한은 백엔드 책임 |
| Capture timestamp 함께 전송 | Android → Spring → AI 서버로 그대로 릴레이 | **백엔드 미팅 안건**: WebSocket 메시지 포맷에 timestamp 필드 추가 필요 |
| Frame 처리 시간 | AI 서버 내부 처리 | Spring은 관여하지 않음 |

> **백엔드 설계 문서와의 동기화 필요**: 본 문서의 3-3절(timestamp 기준)이 확정됨에 따라, Android의 WebSocket binary message 포맷에 capture timestamp가 포함되어야 한다. 메시지 포맷 (header에 8 byte 추가, 또는 별도 metadata frame 등) 결정은 백엔드 WebSocket 설계 문서에서 다룬다.
> 

### 7-3. 프론트엔드 인터페이스

| 본 문서가 정의 | 프론트엔드 처리 방식 |
| --- | --- |
| Capture timestamp 전송 요구 | Android는 카메라 콜백의 system timestamp(예: `SystemClock.elapsedRealtimeNanos()`)를 binary message에 포함하여 전송 |
| JPEG 압축 권고 | 백엔드 메시지 크기 제한(10MB)을 위해 Android에서 frame 압축 후 전송 |

> **JPEG 압축 권고**: 본 단계는 JPEG 입력을 받아 디코딩한다. Android는 카메라 frame을 JPEG로 압축하여 전송하면 WebSocket 메시지 크기가 감소한다. 백엔드 메시지 크기 제한(10MB)을 고려할 때 권장된다. JPEG quality factor에 따른 landmark 안정성은 본 단계가 아닌 2-3-3 (Pose 추출) 단계의 검증 영역이다.

## 8. 임계값 보정 실험 계획

본 문서가 정의한 frame-level 품질 검사 임계값과 정책의 타당성을 실측 데이터로 검증하기 위한 실험. 이 실험은 데모 영상 수집 후 진행한다.

| 실험 항목 | 방법 | 검증 대상 |
| --- | --- | --- |
| 평균 휘도 분포 확인 | 정상 영상 / 어두운 영상에서 평균 휘도 측정 | 휘도 임계 50 타당성 |
| Laplacian variance 분포 확인 | 정상 영상 / 모션 블러 영상에서 분산 측정 | 블러 임계 100 타당성 |
| Frame-level 흔들림 임계 검증 | 카메라 고정 영상 / 흔들림 영상 비교 | SSD 200% 임계 타당성 |
| **SSD 오탐률 측정** | 카메라 고정 + 정상 러닝 영상에서 `frame_unstable` 트리거 빈도 | 4-3절 SSD 한계의 실측 영향 정량화 |
| Frame drop 발생 빈도 측정 | 실시간 스트림 환경에서 frame 도착 간격 분포 | timestamp 기반 처리의 타당성 |
| Capture timestamp vs 수신 시각 차이 측정 | 동일 frame의 두 timestamp 차이 분포 | capture timestamp 기준 사용의 정확도 향상 정량화 |
| 본 단계 처리 시간 측정 | 1080p 입력에서 frame당 처리 시간 측정 | 15ms 추정의 타당성 |
| 본 단계 + Pose 추출 통합 latency | end-to-end frame 처리 시간 측정 | 비동기 파이프라인 / frame skip 전략 필요성 판단 |

> 자세 지표(Foot Strike, Knee Flexion, Trunk Lean) 자체의 분류 임계값 검증 실험은 2-3-4 문서의 검증 계획을 따른다. 본 문서는 frame-level 품질 임계값과 시간축 처리에만 집중한다.
>

## 9. 최종 정리

본 문서는 AI 파트의 영상 전처리와 frame-level 품질 검사의 단일 정답 위치이다.

핵심 결정 사항을 다시 정리하면 다음과 같다.

1. 본 단계는 **MediaPipe Pose 추출 직전**에 동작하며, **픽셀 수준 신호만** 본다. Landmark 기반 검사는 2-3-5가 담당한다.
2. 영상은 **30fps와 720p를 목표 상한선으로 한 cap 정규화**를 적용한다. 초과 입력은 downsample하고, 미만 입력은 인위적으로 보간/업스케일하지 않는다. Downsampling은 30fps timestamp grid 기반의 일반 원칙으로 처리된다.
3. **시간축은 Android capture timestamp 기준**으로 관리하며, AI 서버 수신 시각은 latency 모니터링 용도로만 사용한다. WebSocket 스트림 모드의 frame drop이 발생해도 IC 검출 정확도에 영향이 없다. 본 정책은 백엔드 WebSocket 메시지 포맷에 capture timestamp 필드 추가를 요구한다.
4. **2-3-4의 frame 단위 파라미터**는 effective FPS 기준 ms 단위로 해석되어야 한다. 본 문서는 시간축의 단일 정답 위치로서 이 환산 책임을 후속 모듈에 위임한다.
5. Frame-level 품질 검사는 **3종**: 평균 휘도(조도), Laplacian variance(블러), SSD 또는 optical flow(frame-level 흔들림). SSD는 러너 자체 움직임과 카메라 흔들림 구분에 한계가 있으므로 *"frame-level 이상 변동 신호"* 로 해석되며, 단정적 카메라 흔들림 판정은 stride 단위로 보는 2-3-5가 한다.
6. **품질 미달 frame을 거부하지 않는다.** 품질 플래그를 부여하고 후속 단계가 stride 누적에서 가중치 0으로 처리한다. 다중 플래그 frame의 Pose 추출 생략 정책은 향후 확장 영역.
7. 본 단계 처리 시간은 frame당 약 15ms로 추정되며, 후속 Pose 추출(~33ms)과 합산 시 frame 예산을 초과하므로 **비동기 파이프라인 또는 frame skip 전략을 전제**로 한다.

품질 검사 임계값은 모두 초기 default이며, 데모 영상 기반 실험을 통해 보정한다. 모든 reason code의 사용자 메시지는 2-3-5 문서가 단일 정답 위치다.
