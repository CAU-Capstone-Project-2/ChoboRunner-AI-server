2-3-5. 측정 품질 판단 및 예외 처리 

## 1. 이 문서를 한눈에

### 이 문서가 정의하는 것

이 문서는 **"AI가 추출한 자세 데이터를 사용자에게 결과로 제공해도 되는지 판단하는 단계"** 의 정책을 정의합니다. AI 파트 안에서 이 문서는 다음 세 가지의 **단일 정답 위치(single source-of-truth)** 역할을 합니다.

- **MediaPipe Pose 추출 후의 모든 품질 검사 기준** (Pose 후 visibility, 전신 포함, 측면 구도 등)
- **분석 결과 상태값** (`success` / `low_confidence` / `failed`)의 분기 룰
- **모든 reason code의 정의, 우선순위, 사용자 메시지** (단일 정답 위치)

본 문서는 Pose 추출 후 품질 검사뿐 아니라, **자세 지표 산출 결과를 포함한 최종 상태 결정**까지 담당한다.

### 이 문서가 정의하지 않는 것

다음 항목은 다른 문서에 정의되어 있으니 그쪽을 참조해주세요.

| 알고 싶은 것 | 가야 할 문서 |
| --- | --- |
| 영상 길이/FPS/해상도 같은 메타데이터 임계값은? | **2-3-1** (3장 임계값 표) |
| 영상 화질(블러, 휘도, 흔들림)의 frame-level 검사는? | **2-3-2** (Pose 추출 전 품질 검사) |
| 자세 지표(Foot Strike, Knee Flexion, Trunk Lean) 계산식은? | **2-3-4** (자세 지표 산출) |
| IC 후보 검출 알고리즘은? | **2-3-4** (3장 IC 검출) |
| 응답 JSON 메시지 스펙은? | **2-3-7** (자세 분석 결과 구조화) |
| 사용자 촬영 가이드 메시지(촬영 시작 전)는? | **2-3-1** (4장 촬영 가이드) |

### 다른 파트가 알아두면 좋은 핵심 결정 6가지

이 문서에서 결정된 사항 중, 다른 파트(백엔드, 프론트엔드, 다른 AI 모듈)에 영향을 주는 핵심 정책은 다음과 같습니다.

1. **분석은 분석측 한쪽 다리만 측정한다.** 좌우 분리 측정은 하지 않으며, 결과 메시지에는 단일값으로 응답된다.
2. **다중 인물이 화면에 보여도 즉시 거부하지 않는다.** 추적 안정성 기반 판단이며, 추적이 흔들릴 때만 재촬영을 안내한다.
3. **모든 reason code의 사용자 메시지는 이 문서가 단일 정답 위치다.** 프론트엔드는 이 문서 8장 표를 그대로 사용한다.
4. **`low_confidence`는 결과를 제공하되 재촬영 권장 메시지를 함께 보낸다.** 결과 제공 자체를 막지 않으며, 사용자에게 신뢰도 정보를 함께 전달한다.
5. **사용자에게 보여줄 reason code는 1개로 제한한다.** 내부 평가는 복수 reason code를 다루지만, 응답 메시지의 사용자 노출 부분은 우선순위가 가장 높은 1개만 선택한다.
6. **품질 검사는 stride 단위로 누적된다.** WebSocket 실시간 스트림 모드에서 frame별로 즉시 판정하지 않고, stride가 형성된 후 stride 단위로 평가한다.

## 2. 품질 검사의 두 층 — 어디서 무엇을 검사하나

본 시스템의 품질 검사는 **MediaPipe Pose 추출의 전후 두 층**으로 분리되어 있다. 본 문서는 그중 Pose 추출 후 검사를 담당한다.

```
[Frame 입력]
     │
     ▼
┌──────────────────────────────────┐
│ 2-3-2: Pose 추출 전 검사          │
│  ├ 평균 휘도 (조도)                │
│  ├ Laplacian variance (블러)      │
│  └ Optical flow (frame-level 흔들림)│
└──────────────────────────────────┘
     │
     ▼
[MediaPipe Pose 추출 (2-3-3)]
     │
     ▼
┌──────────────────────────────────┐
│ 2-3-5: Pose 추출 후 검사 (본 문서) │
│  ├ Landmark visibility            │
│  ├ 전신 포함 / 발 잘림              │
│  ├ 측면 촬영 구도                   │
│  ├ 분석 대상자 추적 안정성           │
│  ├ 지표별 변동성                    │
│  └ Stride-level 흔들림              │
└──────────────────────────────────┘
     │
     ▼
[자세 지표 산출 (2-3-4)]
     │
     ▼
┌──────────────────────────────────┐
│ 2-3-5: 최종 상태 결정 (본 문서)    │
│  ├ Reason code 평가                │
│  ├ 사용자 노출용 reason code 선택   │
│  └ success / low_confidence / failed│
└──────────────────────────────────┘
     │
     ▼
[결과 메시지 응답 (2-3-7)]
```

### 두 층으로 나누는 이유

- **Pose 추출 전 검사 (2-3-2)**: 픽셀 수준 신호(휘도, 블러, 흔들림). MediaPipe 호출 없이 빠르게 검사 가능. frame-level 단위로 비싼 Pose 추론을 건너뛰는 1차 필터 역할.
- **Pose 추출 후 검사 (2-3-5, 본 문서)**: landmark 좌표 기반 신호(visibility, 신체 위치 관계, 추적 연속성). MediaPipe 결과가 있어야 검사 가능.

이 분리는 **"자기 자신을 보지 못하는 검사를 같은 단계에서 시도하지 않는다"** 는 원칙에 따른다. 예를 들어 *"전신이 화면에 포함되었는가"* 를 Pose 추출 전 단계에서는 알 수 없다. landmark 좌표 없이 사람이 어디 있는지 모르기 때문이다.

## 3. 분석 대상 측 정책

본 시스템은 측면 단일 카메라 환경에서 **분석측 한쪽 다리만 측정**한다.

### 3-1. 분석측 결정

| 항목 | 정책 |
| --- | --- |
| 결정 시점 | 영상 시작 후 **1.5초 이상 경과** 그리고 **유효 frame 30개 이상 확보** — 두 조건 모두 충족된 시점 |
| 결정 기준 | hip, knee, ankle, heel, foot_index의 visibility 평균이 더 높은 측 |
| 변경 가능 여부 | 한 번 결정되면 분석 종료까지 유지 (시계열 일관성 확보) |
| 출력 | 응답 메시지의 `analysis_side` 필드에 `"left"` 또는 `"right"`로 명시 |

> **AND 조건의 이유**: 시간 기준만 사용하면 effective FPS가 낮은 환경에서 표본이 부족할 수 있고, frame 수 기준만 사용하면 effective FPS 변동에 따라 결정 시점이 흔들린다. 두 조건을 모두 충족시켜 어느 환경에서도 충분한 표본 위에서 분석측이 결정되도록 한다. 유효 frame은 분석측 후보 landmark의 평균 visibility가 임계 통과한 frame을 의미한다.
> 

### 3-2. 좌우 분리 측정을 하지 않는 이유

1. **카메라에서 먼 쪽은 self-occlusion이 발생한다.** 측면 영상에서 반대쪽 다리는 분석 대상 다리에 가려져 visibility가 본질적으로 낮다.
2. **2-3-4 문서의 분석측 선택 정책과 정합한다.** 자세 지표 산출 단계가 이미 한쪽만 보는 가정으로 설계되었다.
3. **사용자 화면 단순화에 기여한다.** 단일값이 UX적으로 명확하다.
4. **좌우 비대칭 분석은 향후 확장 영역이다.** 후면 카메라 또는 양측 측면 카메라 도입 시 의미 있는 분석이 가능하다.

### 3-3. 다른 모듈에 미치는 영향

- **2-3-4 (자세 지표 산출)**: 분석측을 입력으로 받아 한쪽만 계산
- **2-3-7 (응답 메시지)**: `initial_knee_flexion_deg` 단일 필드, `analysis_side` 필드 추가
- **백엔드**: 임시 응답의 `_left_deg` / `_right_deg` 분리 제거 → 단일 필드로 통일

## 4. 분석 대상자 추적 안정성

본 시스템은 한 명의 사용자(분석 대상자) 분석을 전제로 한다. 다만 헬스장 러닝머신 환경의 특성상 주변 인물이 일부 화면에 포함될 수 있으므로, 다음 정책을 적용한다.

### 4-1. 다중 인물 처리 원칙

- **다중 인물이 화면에 보인다는 사실 자체로 분석을 거부하지 않는다.** 이 정책은 2-3-1 문서와 동기화되어 있다.
- **분석 대상자의 추적이 안정적으로 유지되는지** 를 기준으로 판단한다.
- 주변 인물로 인해 분석 대상자가 흐트러지거나 추정 대상이 바뀌었다고 의심되는 경우에만 재촬영을 안내한다.

MediaPipe Pose를 `num_poses = 1`로 설정해도, 이는 **한 명의 포즈만 출력한다**는 의미일 뿐 화면에 한 명만 존재한다는 보장이 아니다. 추정 대상이 frame 사이에 다른 사람으로 바뀔 수 있으므로, 본 시스템은 다음 항목들을 추적 안정성의 신호로 사용한다.

### 4-2. 추적 안정성 신호

| 추적 안정성 신호 | 정량 기준 (default) | 평가 방식 | 비고 |
| --- | --- | --- | --- |
| pelvis 중심 좌표 연속성 | window 잔차가 화면 폭의 10% 미만 | 0.5초 sliding window 중앙값 대비 잔차 | 절대 이동량이 아닌 잔차 기준 |
| 신체 크기(landmark scale) 안정성 | window 평균 대비 변동률 ±15% 이내 | 1초 sliding window 평균 대비 변동률 | 카메라가 고정인데 크기 변동 = 다른 사람 의심 |
| 주요 landmark visibility 유지 | 평균 0.6 이상 | 1초 sliding window 평균 | 전반적으로 추적 가능한 상태 |
| heel/foot_index 좌표 연속성 | mid-stance 구간 잔차가 화면 폭의 8% 미만 | **2-3-4의 IC 전후 자연 변동을 제외**한 mid-stance window 잔차 | IC 전후의 발 좌표 급변은 정상 |
| stride 간격 표준편차 | **중앙값 대비** 편차가 평균의 30% 미만 | 누적 stride 5개의 중앙값 기반 | 절대 표준편차가 아닌 중앙값 기반 편차 |

위 기준은 모두 **초기 default 값**이며, 파일럿 데이터 검증을 통해 보정한다.

### 4-2-1. 평활화 원칙

추적 안정성 신호는 모두 **window 기반 잔차**로 평가한다. 인접 frame 간 절대 이동량을 그대로 사용하면 러닝의 주기적 움직임 자체를 추적 실패로 오탐할 가능성이 있다. 본 문서의 모든 추적 안정성 신호는 다음 원칙을 따른다.

1. **단일 frame 변화량이 아닌 짧은 window 평균 변화량을 본다.** 인접 2 frame의 점프는 정상적인 러닝 동작에서도 발생할 수 있다.
2. **좌표 자체가 아니라 평활화 후 잔차를 본다.** Sliding window의 중앙값 또는 평균을 기준선으로 두고, 실제 좌표가 그 기준선에서 얼마나 벗어났는지를 평가한다.
3. **IC 전후 자연 변동이 큰 landmark는 mid-stance 구간에서 평가한다.** Heel/foot_index는 IC 시점에서 정상적으로 급변한다. 이 구간을 잔차 평가에 포함하면 정상 러닝도 트리거될 수 있다. 따라서 IC 사이 중간 시점(mid-stance)을 평가 윈도우로 사용한다.
4. **Stride 간 통계는 표준편차가 아닌 중앙값 기반 편차를 사용한다.** 표준편차는 단일 outlier에 민감하므로, 정상 러닝의 일시적 변동 1회가 전체 stride 분산을 키울 수 있다. 중앙값 대비 편차는 outlier의 영향을 작게 받는다.

### 4-3. 추적 안정성 신호 → Reason Code 매핑 규칙

추적 안정성 신호가 어떻게 조합될 때 어느 reason code를 트리거하는지 명시한다. 본 절은 구현 단계에서 분기 로직 작성에 직접 사용된다.

| Reason code | 트리거 조건 (신호 조합) | 강도 |
| --- | --- | --- |
| `target_switch_detected` | pelvis 중심 잔차 급증 (>15%) **AND** scale 급변 (±20% 이상) **AND** visibility 평균 일시 붕괴 (<0.4) — 모두 동시 발생 | failed |
| `target_lost` | visibility 평균이 5초 이상 연속으로 0.4 미만 유지 (대상자 자체가 보이지 않음) | failed |
| `unstable_landmark_sequence` | landmark 좌표 잔차 임계 위반(heel/foot mid-stance 잔차 >8%)이 분석 시간의 30% 이상에서 발생, 단 pelvis 중심 + scale은 안정 | low_confidence |
| `background_person_interference` | 추적 안정성은 유지되지만 visibility가 borderline (0.4~0.6) 구간에서 30% 이상 발생 + scale에 미세 변동 (간섭 의심) | low_confidence |

> **운영 원칙**: 위 4개 reason code는 **상호 배타적**이 아니다. 동시에 활성화될 수 있다. 다만 사용자에게 보여줄 메시지는 본 문서 8-7절의 우선순위 규칙에 따라 1개로 좁힌다.
> 

### 4-4. 신호 평가 윈도우

각 신호는 다음 윈도우에서 평가된다.

| 신호 종류 | 평가 윈도우 |
| --- | --- |
| Pelvis 중심 잔차 | 0.5초 sliding window |
| Visibility 평균 | 최근 1초 (30 frame 기준) sliding window |
| Scale 변동 | 최근 1초 sliding window |
| heel/foot 잔차 | mid-stance 구간 (IC 사이 중간 시점) |
| Stride 간격 분산 | 누적 stride 5개의 중앙값 대비 편차 |

## 5. Pose 추출 후 품질 검사 항목

본 장은 MediaPipe Pose 추출이 완료된 후 수행하는 품질 검사 항목을 정의한다. 본 장의 모든 임계값은 **초기 default 값**이며, 파일럿 데이터 검증을 통해 보정한다.

### 5-1. Landmark visibility 검사

| 검사 항목 | 임계값 (default) | 미충족 시 reason code |
| --- | --- | --- |
| 분석측 hip, knee, ankle visibility 평균 | 0.6 이상 | `lower_body_not_visible` |
| 분석측 heel, foot_index visibility 평균 | 0.6 이상 | `foot_not_visible` |
| 양측 shoulder visibility 평균 | 0.6 이상 | `upper_body_not_visible` |
| 전체 주요 landmark visibility 평균 | 0.5 이상 | `low_landmark_visibility` |
| 유효 frame 비율 (visibility 임계 통과) | 60% 이상 | `low_landmark_visibility` |

### 5-2. 전신 포함 검사

| 검사 항목 | 임계값 (default) | 미충족 시 reason code |
| --- | --- | --- |
| nose, ankle 모두 visibility 0.6 이상 frame 비율 | 60% 이상 | `body_not_fully_visible` |
| 머리부터 발끝까지 모든 주요 landmark가 화면 좌표 [0, 1] 범위 내 | 60% 이상 | `body_not_fully_visible` |

> **2-3-2와의 차이**: *"전신이 보이는가"* 는 landmark 좌표를 기반으로 판단하므로 Pose 추출 후 검사다. 픽셀 수준 화질 검사(블러/휘도/흔들림)는 2-3-2가 담당한다.
> 

### 5-3. 발 잘림 검사

| 검사 항목 | 임계값 (default) | 미충족 시 reason code |
| --- | --- | --- |
| 분석측 ankle, heel, foot_index의 y 좌표 | 0.95 미만 (화면 하단에 닿지 않음) | `foot_out_of_frame` |
| 위 조건을 만족하는 frame 비율 | 60% 이상 | `foot_out_of_frame` |

> **원리**: 화면 좌표는 정규화 좌표 [0, 1]을 사용한다. y > 0.95는 사실상 화면 경계에 닿은 상태로, 발이 잘렸을 가능성이 높다.
> 

### 5-4. 측면 촬영 구도 검사

측면 촬영이 정확히 이루어졌는지를 **1차 조건 + 보조 조건**으로 검사한다. 단일 ratio만 사용하면 측면에 가까울수록 분모가 작아져 ratio가 불안정해질 수 있고, shoulder는 팔 위치/상체 회전/카메라 높이에 따라 흔들리기 때문에 보조 조건으로 robustness를 확보한다.

| 조건 종류 | 검사 항목 | 임계값 (default) |
| --- | --- | --- |
| **1차 조건 (필수)** | 좌우 hip의 x 좌표 거리 | 화면 폭의 5% 미만 |
| **보조 조건 (둘 중 하나 이상 충족)** | (a) 좌우 shoulder의 x 좌표 거리 | 화면 폭의 7% 미만 |
|  | (b) torso yaw proxy: 좌우 hip 거리와 hip-shoulder 수직 거리의 비율 | 0.15 미만 |

판정 규칙:

| 조건 충족 상태 | 판정 |
| --- | --- |
| 1차 조건 충족 + 보조 조건 (a) 또는 (b) 중 1개 이상 충족 | 측면 구도 정상 |
| 1차 조건만 충족, 보조 조건 모두 위반 | `invalid_view` (`low_confidence` 트리거) |
| 1차 조건 위반 (충족 frame 비율이 60% 미만) | `invalid_view` (`failed` 트리거) |

> **원리**: 정면을 바라볼수록 좌우 hip의 가로 거리가 커지고, 측면이면 좌우 hip이 거의 겹쳐서 거리가 0에 수렴한다. 다만 hip 거리만으로는 상체 비틀림(shoulder 회전)을 감지하지 못하므로, shoulder 거리 또는 torso yaw proxy를 보조로 둔다. Yaw proxy는 hip-shoulder 수직 거리(키에 비례하는 안정 척도) 대비 hip 가로 거리의 비율로, 카메라와의 거리에 무관한 정규화된 측면 정도 지표이다.
> 

### 5-5. 카메라 흔들림 검사 (Stride-level)

frame-level 흔들림 검사는 2-3-2가 담당하지만, stride 단위에서 누적된 흔들림은 본 문서가 평가한다. 본 검사는 본 문서 안에서 자기충족적으로 정의된다.

| 검사 항목 | 임계값 (default) | 미충족 시 reason code |
| --- | --- | --- |
| pelvis_x의 stride 평균 대비 변동 | ±30% 이내 (2-3-4 문서 9장 기준 인용) | `camera_unstable` |

> **2-3-2와의 차이**: 2-3-2는 frame 간 optical flow magnitude로 즉각적 흔들림을 검사하고, 본 절은 stride 단위로 누적된 평균 흔들림을 본다. 짧은 순간의 흔들림은 frame-level에서 잡히고, 분석 전체에 걸친 카메라 불안정은 stride-level에서 잡힌다.
> 

### 5-6. 지표별 변동성 검사

같은 영상 안에서 측정된 자세 지표가 stride마다 비정상적으로 흔들리면, 측정값 자체의 신뢰도가 낮다고 판단한다.

| 지표 | 변동성 기준 (default) | 미충족 시 reason code |
| --- | --- | --- |
| Foot angle | stride 간 표준편차 < 5° | `unstable_foot_angle` |
| Knee flexion angle | stride 간 표준편차 < 7° | `unstable_knee_angle` |
| Trunk lean angle | stride 간 표준편차 < 4° | `unstable_trunk_angle` |

> **변동성 기준 vs 임계 구간 폭**: Foot angle 분류 임계 구간이 ±5°(2-3-4 문서 4장)인데, 측정 변동성이 그 폭에 가까우면 분류 자체가 의미가 없어진다. 변동성 기준을 임계 구간보다 작게 설정한 이유다. 위 임계값은 모두 초기 default이며, 정상 러닝 영상의 stride 간 분산 분포를 확인하여 보정한다.
> 

### 5-7. IC 후보 검증

자세 지표 산출이 가능하려면 충분한 stride가 검출되어야 한다. IC 후보 자체의 검증은 2-3-4가 담당하며, 본 문서는 산출 가능 여부만 판단한다.

| 검사 항목 | 임계값 (default) | 미충족 시 reason code |
| --- | --- | --- |
| 전체 IC 후보 수 | 3개 이상 (2-3-4 문서 8장 인용) | `insufficient_stride` |
| 신뢰도 `high`/`medium`인 IC 수 | 2개 이상 | `low_ic_confidence` |
| Trunk Lean 계산용 window 확보 | IC ± 2~3 frame이 visibility 임계 통과 | `insufficient_window` |

## 6. 분석 상태값 분기 룰

본 시스템의 분석 상태는 **3가지뿐**이다. 부분 결과 제공은 1차 MVP에서 제공하지 않는다(2-3-1 문서 2-4절 인용).

### 6-1. 상태값 정의

| 상태 | 의미 | 사용자 처리 |
| --- | --- | --- |
| `success` | 핵심 지표 3개 모두 정상 계산, 모든 품질 임계 통과 | 최종 분석 결과 제공 |
| `low_confidence` | 핵심 지표 3개 모두 계산은 가능하나 일부 품질 기준이 임계에 가까움 | **결과 제공 + 재촬영 권장 메시지 동시 제공** |
| `failed` | 핵심 지표 중 하나 이상 계산 불가 또는 심각한 품질 문제 | 분석 결과 미제공, 재촬영 안내 |

> **`low_confidence` 정책 명확화**: `low_confidence`에서 사용자에게 결과를 보여줄지 말지에 대한 결정. 본 시스템은 결과를 함께 제공하고 재촬영 권장 메시지를 함께 표시한다. 이유: (1) 사용자 입장에서 분석을 끝까지 기다린 후 결과를 못 보는 경험은 부정적, (2) 신뢰도 정보(분산, IQR 등)를 함께 제공하므로 사용자가 결과를 어느 정도 신뢰해야 할지 판단 가능, (3) 2-3-7 응답 메시지 스펙이 이미 `low_confidence`에서도 `metrics`/`metric_details`/`message`를 모두 응답하는 구조로 설계되어 있다.
> 

### 6-2. 분기 룰

```
[모든 검사 통과 + IC ≥ 3개 + 변동성 임계 내]
                 → success

[일부 검사 borderline + 모든 핵심 지표 계산 가능]
                 → low_confidence

[지표 1개 이상 계산 불가]
                 → failed

[심각한 품질 문제 (대상자 추적 실패, 영상 메타데이터 미달 등)]
                 → failed
```

### 6-3. 상태값 결정 우선순위

여러 reason code가 동시에 활성화될 수 있다. 다음 우선순위로 최종 상태를 결정한다.

| 우선순위 | 조건 | 결과 |
| --- | --- | --- |
| 1 | `failed` 트리거 reason code가 1개 이상 | `failed` |
| 2 | `low_confidence` 트리거 reason code가 1개 이상 | `low_confidence` |
| 3 | 모든 검사 통과 | `success` |

각 reason code의 트리거 강도(`failed`인지 `low_confidence`인지)는 8장 표에 정의되어 있다.

> **2-3-1과의 정합**: 2-3-1 문서의 메타데이터 검증 우선순위 규칙(`failed` > `low_confidence` > `success`)과 동일하게 적용된다. 즉, 메타데이터 검증 결과와 본 문서의 검증 결과를 합칠 때도 어느 한 단계에서든 `failed`가 발생하면 최종 상태는 `failed`이다.
>

## 7. 실시간 스트림 모드에서의 점진적 누적 분석

본 시스템은 WebSocket 기반 실시간 스트림으로 frame을 수신한다(2-3-1 문서 2장). 분석은 frame이 누적되면서 점진적으로 수행된다.

### 7-1. 누적 분석 단계

| 누적 시점 | 분석 상태 |
| --- | --- |
| Frame 누적 시작 ~ 첫 IC 검출 전 | "분석 준비 중" (응답 없음 또는 진행 상태 메시지) |
| 첫 IC 검출 ~ 유효 stride 3개 누적 전 | "분석 중" (응답 없음 또는 진행 상태 메시지) |
| 유효 stride 3개 누적 후 | 매 stride마다 누적 결과 갱신 (frame별 즉시 응답은 별도) |
| 사용자 분석 종료 또는 frame 수신 중단 | 최종 누적 결과 응답 (`success` / `low_confidence` / `failed`) |

### 7-2. 진행 상태 vs 최종 상태 — 명확 구분

본 시스템에서 응답 메시지는 두 종류로 분리된다.

| 응답 종류 | 발생 시점 | 본 문서의 상태값 적용 여부 |
| --- | --- | --- |
| **frame별 즉시 응답** | 각 frame 추론 완료 시점 | 적용 안 함 (frame별 응답은 본 문서의 status 필드를 가지지 않음) |
| **최종 누적 응답** | 분석 종료 trigger 발생 시점 | 본 문서의 모든 품질 검사가 적용되어 `success` / `low_confidence` / `failed` 결정 |

이 분리는 사용자 경험 일관성을 위해 필수적이다. frame별로 즉시 status가 갱신되면, 분석 도중 화면이 *"분석 중 → 분석 실패 → 분석 중 → 분석 완료"* 처럼 흔들릴 수 있다. 본 문서가 정의하는 모든 품질 검사 기준은 **최종 누적 응답 시점에만 적용**된다.

분석 진행 상황을 사용자에게 알릴 필요가 있을 때는 2-3-7 문서의 `analysis_progress` 메시지로 별도 처리하며, 이 진행 메시지는 본 문서의 status 필드와 독립이다.

## 8. Reason code 사전

본 장은 본 시스템의 **모든 reason code의 단일 정답 위치**이다. 다른 문서에서 reason code를 인용할 때는 본 장의 표를 직접 참조해야 한다.

> **확장 여지**: 1차 MVP에서는 reason code = 사용자 메시지의 1:1 매핑이지만, 향후 다국어 지원이나 A/B 테스트가 필요해지면 이 표를 internal code 사전과 user-facing message 카탈로그로 분리할 수 있다. 이 경우 본 문서는 internal code 사전을 유지하고, message 카탈로그는 별도 i18n 파일로 분리한다. 본 절의 컬럼 구조(코드 / 의미 / 사용자 메시지)는 향후 분리를 염두에 두고 설계되었다.
>

| `unstable_trunk_angle` | Trunk lean stride 간 표준편차 4° 초과 | 상체 자세가 매번 달라 안정적인 측정이 어려웠습니다. | low_confidence |

### 8-7. Reason code 우선순위 체계

여러 reason code가 동시에 활성화될 수 있다. 내부 평가에서는 모든 활성 코드를 추적하지만, 사용자에게 보여줄 메시지는 1개로 좁힌다.

### 8-7-1. 사용자 노출 reason code 선택 원칙

- **failed 그룹이 항상 low_confidence 그룹보다 우선이다.** 한 번이라도 failed가 트리거되면 사용자에게는 failed 그룹에서 선택된 메시지가 표시된다.
- **그룹 내에서는 사용자가 행동 변화로 즉시 해결 가능한 코드가 더 추상적인 코드보다 우선이다.** 예를 들어 `foot_out_of_frame`("발이 잘림 → 카메라 멀리")이 `low_landmark_visibility`("신뢰도 낮음")보다 사용자에게 더 유용하다.

### 8-7-2. failed 그룹 우선순위

| 그룹 | Reason codes (그룹 내 우선순위 순) |
| --- | --- |
| 1. 분석 대상자 추적 실패 | `target_switch_detected` > `target_lost` |
| 2. 발 가시성 실패 | `foot_out_of_frame` > `foot_not_visible` |
| 3. 전신 가시성 실패 | `body_not_fully_visible` |
| 4. 측면 구도 실패 | `invalid_view` (failed 강도) |
| 5. 메타데이터/IC 실패 | `too_short` > `low_resolution` > `low_fps` > `insufficient_stride` |

### 8-7-3. low_confidence 그룹 우선순위

| 그룹 | Reason codes (그룹 내 우선순위 순) |
| --- | --- |
| 6. 추적 borderline | `unstable_landmark_sequence` > `background_person_interference` |
| 7. 측면 구도 borderline | `invalid_view` (low_confidence 강도) |
| 8. 일반 품질 저하 | `camera_unstable` > `low_landmark_visibility` > `lower_body_not_visible` > `upper_body_not_visible` |
| 9. IC 신뢰도 부족 | `low_ic_confidence` > `insufficient_window` |
| 10. 지표 변동성 | `unstable_foot_angle` > `unstable_knee_angle` > `unstable_trunk_angle` |

> **그룹 간 우선순위**: failed 그룹(1~5)이 항상 low_confidence 그룹(6~10)보다 우선. failed 그룹 내에서는 1번 그룹이 가장 우선이며 5번이 가장 낮다. low_confidence 그룹도 마찬가지로 6번이 가장 우선이며 10번이 가장 낮다.
> 

### 8-7-4. 적용 예시

| 활성화된 reason codes (내부) | 사용자 노출 reason code | 사용자 메시지 |
| --- | --- | --- |
| `foot_out_of_frame`, `foot_not_visible`, `low_landmark_visibility` | `foot_out_of_frame` | 발이 화면 아래에서 잘립니다. 카메라를 조금 멀리 두세요. |
| `target_switch_detected`, `low_landmark_visibility` | `target_switch_detected` | 분석 대상자가 다른 사람과 겹쳐 자세 추정이 불안정합니다. 다시 촬영해주세요. |
| `body_not_fully_visible`, `low_landmark_visibility` | `body_not_fully_visible` | 머리부터 발끝까지 전신이 화면에 들어오도록 다시 촬영해주세요. |
| `unstable_foot_angle`, `unstable_knee_angle`, `low_ic_confidence` | `low_ic_confidence` | 착지 시점을 안정적으로 찾기 어렵습니다. 다시 촬영해주세요. |
| `camera_unstable`, `unstable_landmark_sequence` | `unstable_landmark_sequence` | 자세 추정이 일부 구간에서 흔들렸습니다. 가능하면 다시 촬영해주세요. |

> **선택 원칙 재확인**: 동일 그룹 내에서는 **사용자가 행동 변화로 즉시 해결 가능한 코드**가 위에 위치한다. 예를 들어 `foot_out_of_frame`("발 잘리지 않게 카메라 멀리")이 `low_landmark_visibility`("신뢰도 낮음")보다 사용자에게 즉시 도움된다.

## 10. 품질 기준 보정 실험 계획

본 문서가 정의한 품질 검사 임계값의 타당성을 실측 데이터로 검증하기 위한 실험. 이 실험은 데모 영상 수집 후 진행한다.

| 실험 항목 | 방법 | 검증 대상 |
| --- | --- | --- |
| Visibility 0.6 임계 타당성 | 정상 영상에서 주요 landmark visibility 분포 확인 | 본 문서 5-1 임계값 |
| 측면 구도 1차/보조 조건 임계 타당성 | 측면/사선/정면 영상 비교, hip 거리와 shoulder 거리/yaw proxy 분포 확인 | 본 문서 5-4 임계값 |
| 발 잘림 검출 정확도 | 의도적으로 발 잘린 영상 테스트 | 본 문서 5-3 검출 |
| 변동성 기준 타당성 | 정상 영상에서 stride 간 표준편차 분포 확인 | 본 문서 5-6 임계값 |
| 추적 안정성 신호 검증 | 주변 인물 간섭 영상 / 대상자 전환 시뮬레이션 영상 테스트 | 본 문서 4-3 매핑 규칙 |
| **추적 안정성 평활화 효과 검증** | 동일 영상에서 절대값 기준 vs window 잔차 기준의 트리거 빈도 비교 | 본 문서 4-2-1 평활화 원칙의 오탐 감소 효과 |
| Reason code 우선순위 적합성 | 다양한 실패 케이스 수집 후 사용자 표시 메시지가 적절한지 확인 | 본 문서 8-7 우선순위 표 |
| `low_confidence` 케이스 사용자 반응 | 결과 + 재촬영 권장 메시지 노출 시 사용자 행동 관찰 | 6-1 정책 적합성 |

## 11. 최종 정리

본 문서는 AI 파트의 측정 품질 판단 기준과 모든 reason code 정의의 단일 정답 위치이다.

핵심 결정 사항을 다시 정리하면 다음과 같다.

1. 품질 검사는 **MediaPipe Pose 추출의 전후 두 층으로 분리**되며, 본 문서는 Pose 추출 후 검사부터 최종 상태 결정까지 담당한다.
2. 분석은 **분석측 한쪽 다리만 단일값으로 측정**하며, 좌우 분리 측정은 하지 않는다. 분석측은 **1.5초 이상 경과 AND 30 frame 이상 확보** 시점에 결정된다.
3. 다중 인물이 화면에 보여도 **즉시 거부하지 않으며**, 추적 안정성 기반으로 판단한다. 추적 안정성 신호는 모두 **window 기반 평활화 후 잔차**로 평가한다.
4. 측면 구도 검사는 **1차 조건(hip 거리) + 보조 조건(shoulder 거리 또는 yaw proxy)** 조합으로 robust하게 판정한다.
5. 분석 상태는 **`success` / `low_confidence` / `failed` 3가지뿐**이며, `low_confidence`는 결과를 제공하되 재촬영 권장 메시지를 함께 표시한다. 부분 결과는 1차 MVP에서 제공하지 않는다.
6. 모든 **reason code의 사용자 메시지**는 본 문서 8장이 단일 정답 위치이며, **사용자 노출 reason code는 우선순위에 따라 1개로 좁힌다**.
7. WebSocket 실시간 스트림 모드에서는 **점진적 누적 분석**이 수행되며, 본 문서의 상태값은 최종 누적 결과 응답 시점에만 적용된다. frame별 즉시 응답은 본 문서의 status 필드와 독립이다.

품질 기준 임계값은 모두 초기 default이며, 데모 영상 기반 실험을 통해 보정한다.
