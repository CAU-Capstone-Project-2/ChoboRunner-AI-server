2-3-3. Pose Landmark 추출

## 1. 이 문서를 한눈에

### 이 문서가 정의하는 것

이 문서는 **"MediaPipe Pose를 호출하여 정규화 frame에서 신체 landmark를 추출하는 모듈"** 의 정책을 정의합니다. AI 파트 안에서 이 문서는 다음 두 가지의 **단일 정답 위치(single source-of-truth)** 역할을 합니다.

- **MediaPipe Pose 사용 정책** (모델 종류, 파라미터, 추출 옵션, 실행 모드)
- **Landmark 추출 결과의 출력 구조** (frame별 landmark 리스트 형식)

### 이 문서가 정의하지 않는 것

다음 항목은 다른 문서에 정의되어 있으니 그쪽을 참조해주세요.

| 알고 싶은 것 | 가야 할 문서 |
| --- | --- |
| 어떤 landmark를 사용해서 어떤 자세 지표를 계산하나요? | **2-3-4** (자세 지표 산출) |
| Visibility 임계값(0.6) 등 품질 기준은? | **2-3-5** (Pose 추출 후 품질 검사) |
| Frame-level 품질이 낮은 frame은 어떻게 처리하나요? | **2-3-2** (Frame-level 품질 검사) |
| Frame timestamp는 어떻게 부여되나요? | **2-3-2** (3-3절 capture timestamp 정책) |
| 분석측 한쪽 다리만 사용한다는 정책은? | **2-3-5** (3장 분석 대상 측 정책) |
| EMA smoothing, outlier 처리는? | **2-3-4** (3-5절 좌표 안정화) |

### 다른 파트가 알아두면 좋은 핵심 결정 5가지

이 문서에서 결정된 사항 중, 다른 파트(AI 모듈)에 영향을 주는 핵심 정책은 다음과 같습니다.

1. **MediaPipe Pose Tasks API의 Pose Landmarker를 사용한다.** 이전 세대인 MediaPipe Pose Solution API는 사용하지 않는다.
2. **운영 환경은 Live stream mode, 검증/실험 환경은 Video mode를 사용한다.** WebSocket 실시간 스트림 환경에서는 Live stream mode가 필수이며, 파일럿 데이터 검증 등 오프라인 batch 처리는 Video mode를 사용한다.
3. **`num_poses = 1`로 단일 인물 모드 사용.** 한 명의 pose만 반환되며, 이 pose가 항상 분석 대상자라는 보장은 없다. 분석 대상자 전환 여부 판단은 2-3-5의 추적 안정성 정책으로 위임한다.
4. **본 모듈은 모든 frame에 대해 추출을 시도한다.** Frame-level 품질 플래그가 있는 frame이라도 추출 시도는 수행하며, 실패 여부는 결과에 명시한다. 사용 여부는 후속 모듈이 판단한다.
5. **출력 좌표는 normalized 좌표(x, y ∈ [0, 1])를 기본으로 사용한다.** World landmark(3D 좌표)는 본 v1에서 사용하지 않는다.

## 2. 본 단계의 위치

본 단계는 **영상 전처리(2-3-2) 직후, 자세 지표 산출(2-3-4) 직전**에 동작한다.

```
[Android 카메라]
     │ WebSocket binary frame + capture timestamp
     ▼
[Spring 백엔드]   ── 단순 릴레이
     │
     ▼
[2-3-2 영상 전처리 + Frame-level 품질 검사]
     │ 30fps × 720p 정규화 frame + capture timestamp + 품질 플래그
     ▼
┌─────────────────────────────────────┐
│ 본 문서 담당                          │
│  ├ MediaPipe Pose 호출               │
│  │   (Live stream mode 운영,         │
│  │    Video mode는 검증/실험)         │
│  ├ Landmark 추출                     │
│  ├ Timestamp 그대로 후속 모듈로 전달 │
│  └ Pose 추출 결과 + visibility 전달   │
└─────────────────────────────────────┘
     │
     ▼
[2-3-4 자세 지표 산출]
     │
     ▼
[2-3-5 측정 품질 판단]
     │
     ▼
[2-3-7 결과 메시지 응답]
```

### 본 단계의 책임 경계

| 본 단계 (2-3-3) | 다른 단계 |
| --- | --- |
| MediaPipe Pose 호출 자체 | (다른 모듈 모두) |
| Landmark 좌표 + visibility 추출 | — |
| Frame별 추출 성공/실패 표시 | 후속 모듈이 사용 |
| Timestamp passthrough | **2-3-2 (3-3절 capture timestamp 정책)** |
| Smoothing, EMA, outlier 제거 | **2-3-4 (3-5절 좌표 안정화)** |
| Visibility 임계값 평가 | **2-3-5 (5-1절 visibility 검사)** |
| 분석측 결정 | **2-3-5 (3장 분석 대상 측 정책)** |

본 단계는 **MediaPipe 호출과 결과 전달까지만** 담당하며, 추출된 landmark의 후처리(smoothing, 분석측 선택, 품질 판단)는 모두 다른 문서가 담당한다.

## 3. MediaPipe Pose 사용 정책

### 3-1. 사용 API

| 항목 | 정책 |
| --- | --- |
| API | **MediaPipe Pose Tasks API** (Pose Landmarker) |
| 사용 안 함 | MediaPipe Pose Solution API (이전 세대) |

> **Tasks API를 선택한 이유**: MediaPipe의 신세대 API로, 모델 다운로드/캐싱 정책이 명확하고 platform 통합이 표준화되어 있다. Tasks API는 향후 모델 업데이트와 추가 기능(landmark 정확도 옵션 등) 적용이 더 안정적이다. 또한 Live stream mode 지원이 Tasks API에 일관되게 통합되어 있어 실시간 스트림 처리에 적합하다.
> 

### 3-2. 모델 선택

| 항목 | 정책 |
| --- | --- |
| **기본 후보** | **Pose Landmarker `full` 모델** — 33개 landmark 추출, 정확도 우선 |
| **대체 후보** | `lite` 모델 — 정확도 trade-off로 약 50% 빠른 추론 |
| **최종 선택 시점** | 데모 영상 기반 파일럿 실험 후 본 문서 default로 확정 |

### 모델 선택 기준 (구현 단계 실험 후 확정)

본 시스템은 발끝(foot_index)과 발뒤꿈치(heel)의 정확도가 핵심이므로(Foot Strike Pattern, IC 검출의 직접 입력), 다음 3가지 기준으로 두 모델을 비교 평가한다.

| 평가 기준 | 측정 방식 | 의미 |
| --- | --- | --- |
| Heel / foot_index visibility 안정성 | 정상 영상에서 두 landmark의 visibility 평균 및 표준편차 | 모델이 발 부위를 얼마나 안정적으로 추적하는가 |
| `pose_detected = True` 비율 | 전체 frame 중 추출 성공 frame 비율 | 모델이 사람을 얼마나 robust하게 검출하는가 |
| Frame당 처리 시간 | 30fps frame당 평균 추론 시간 (ms) | 실시간 처리 가능성 |

### 선택 결과 적용

세 기준 중 visibility 안정성이 가장 우선되며, 처리 시간은 5장의 비동기 파이프라인 / adaptive inference 전제 하에서 부차적 기준으로 본다. 즉, **`lite`가 처리 시간에서 우월해도 visibility 안정성에서 크게 뒤지면 `full`을 선택**한다. 최종 선택된 모델을 본 문서 default로 확정한다.

### 3-3. 추출 파라미터 및 실행 모드

### 기본 파라미터

| 파라미터 | 기본값 | 비고 |
| --- | --- | --- |
| `num_poses` | **1** | 단일 인물 모드 |
| `min_pose_detection_confidence` | 0.5 (default) | MediaPipe 기본값 사용 |
| `min_pose_presence_confidence` | 0.5 (default) | MediaPipe 기본값 사용 |
| `min_tracking_confidence` | 0.5 (default) | MediaPipe 기본값 사용 |
| `output_segmentation_masks` | False | 본 v1에서 segmentation은 사용 안 함 |

### `num_poses = 1`의 정확한 의미

`num_poses = 1`은 **한 명의 pose만 반환**한다는 의미이며, **이 pose가 항상 분석 대상자라는 보장은 없다.** 화면에 여러 명이 있을 때 MediaPipe가 어느 한 명을 선택하는 기준은 모델 내부 휴리스틱에 따르며, 우리가 의도한 분석 대상자와 다를 수 있다.

따라서 분석 대상자 전환 여부 판단은 본 모듈이 아니라 **2-3-5의 추적 안정성 정책** (`target_switch_detected`, `target_lost` reason code)으로 위임한다. 본 모듈은 단순히 MediaPipe가 반환한 pose를 그대로 후속 모듈에 전달하며, 그 pose가 분석 대상자인지 여부는 판단하지 않는다.

### 실행 모드 분리

본 모듈은 두 가지 실행 모드를 사용 환경에 따라 선택한다.

| 모드 | 사용 환경 | 특징 | 비고 |
| --- | --- | --- | --- |
| **Live stream mode (운영)** | WebSocket 실시간 스트림 환경 | 비동기 callback 기반, frame timestamp가 필수 인자 | MVP 운영 환경의 기본 모드 |
| **Video mode (검증/실험)** | 저장 영상 파일 기반 batch 처리 | 동기 호출, 영상 전체를 순차 처리 | 파일럿 데이터 검증, 임계값 보정 실험 등에 사용 |

운영 환경에서는 **Live stream mode가 필수**이다. WebSocket으로 도착하는 frame은 비동기적으로 처리되어야 하며, frame timestamp가 정확하게 추적되어야 한다. Video mode는 오프라인 검증용으로만 사용하며, 운영 코드 경로와 분리된 별도 entry point로 구현한다.

### 3-4. 좌표 형식

본 v1에서는 **normalized landmark(2D 정규화 좌표)** 를 사용한다.

| 종류 | 사용 여부 | 설명 |
| --- | --- | --- |
| Normalized landmarks | **사용** | x, y ∈ [0, 1], visibility ∈ [0, 1] |
| World landmarks | 사용 안 함 | 3D 좌표 (m 단위). 향후 확장 시 검토 항목 |

> **World landmark를 사용하지 않는 이유**: World landmark는 카메라가 보는 시점에서의 3D 추정이며, 2D 픽셀 기반 측정과 절대값이 다르다. 본 시스템의 자세 지표는 모두 영상 좌표계 기반으로 정의되어 있으므로(2-3-4 3-2절), normalized 좌표만 사용한다. World landmark는 향후 3D 분석 확장 시 도입 예정이다.
> 

### 3-5. Timestamp 처리 정책

본 모듈은 2-3-2에서 부여한 `timestamp_sec`(Android capture timestamp 기준)를 그대로 후속 모듈에 전달한다.

| 항목 | 정책 |
| --- | --- |
| Timestamp 출처 | 2-3-2가 부여한 capture timestamp (단일 정답 위치는 2-3-2 문서 3-3절) |
| MediaPipe 호출 시 | Live stream mode에서는 `timestamp_ms`가 필수 인자이므로, 2-3-2가 부여한 capture timestamp를 ms로 변환하여 전달 |
| 후속 모듈로의 전달 | 입력 frame의 timestamp를 출력 구조에 그대로 보존 (frame_index보다 timestamp 우선) |
| Timestamp 변환 / 보정 | **본 모듈은 timestamp를 변경하지 않는다.** 단순 passthrough |

> **Timestamp의 단일 정답 위치**: Frame timestamp가 무엇을 의미하는지(capture vs 수신 시각)는 2-3-2 문서 3-3절이 단일 정답 위치이다. 본 모듈은 이 정책을 따르며, capture timestamp 누락 시 fallback 처리도 2-3-2의 정책을 그대로 따른다 (`timestamp_fallback` 플래그 보존).
> 

### Live stream mode callback 결과 순서

MediaPipe Tasks API의 Live stream mode는 비동기 callback 방식으로 결과를 반환한다. **Callback 결과가 frame 도착 순서대로 반환된다는 보장이 없으므로**, 본 모듈은 다음 정책으로 운영한다.

- 출력 구조에 항상 `frame_index`와 `timestamp_sec`를 포함하여 후속 모듈이 정렬에 사용할 수 있도록 한다.
- 후속 모듈(2-3-4 등)은 frame_index보다 **timestamp_sec를 정렬 키로 우선**한다.
- 본 모듈 내부에서 결과 순서를 강제로 정렬하지 않으며, 정렬 책임은 후속 모듈이 진다.

## 4. 추출 결과 출력 구조

본 단계는 frame별로 다음 구조를 후속 단계(2-3-4)에 전달한다.

```python
{
    "frame_index": 42,                  # 도착 순서 (2-3-2에서 부여)
    "timestamp_sec": 1.400,              # capture timestamp 기준 (2-3-2 부여, 본 모듈 passthrough)
    "pose_detected": True,               # MediaPipe 추출 성공 여부
    "landmarks": {                       # pose_detected = True일 때만 채워짐 (운영 모드: 6종)
        "shoulder": {
            "left":  {"x": 0.42, "y": 0.31, "visibility": 0.95},
            "right": {"x": 0.58, "y": 0.31, "visibility": 0.93}
        },
        "hip": { ... },
        "knee": { ... },
        "ankle": { ... },
        "heel": { ... },
        "foot_index": { ... }
    },
    "landmarks_full": { ... },           # 디버그 모드에서만 채워짐 (33개 전체 landmark)
    "frame_quality_flags": []            # 2-3-2에서 받은 품질 플래그 그대로 전달
}
```

### 4-1. 사용하는 landmark 6종

자세 지표 계산에 필요한 6종의 landmark만 후속 단계로 전달한다. MediaPipe가 추출하는 33개 전체를 모두 전달하지 않는다.

| Landmark | MediaPipe Index | 사용 지표 |
| --- | --- | --- |
| shoulder | 11 (left), 12 (right) | Trunk Lean (좌우 평균점) |
| hip | 23 (left), 24 (right) | Trunk Lean, Initial Knee Flexion, IC 검출 (pelvis_x) |
| knee | 25 (left), 26 (right) | Initial Knee Flexion |
| ankle | 27 (left), 28 (right) | Initial Knee Flexion, Foot Strike Pattern |
| heel | 29 (left), 30 (right) | Foot Strike Pattern, IC 검출 |
| foot_index | 31 (left), 32 (right) | Foot Strike Pattern, IC 검출 |

> **6종으로 제한하는 이유**: 분석에 사용하지 않는 landmark까지 전달하면 모듈 간 인터페이스가 비대해지고, 변경 비용이 커진다. 향후 추가 지표가 도입되면 본 절을 확장한다.
> 

### 디버그 모드 예외

운영 환경에서는 6종만 전달하지만, **디버깅/시각화/로그 분석 시 전체 33개 landmark가 필요한 경우**가 있다.

| 활용 사례 | 필요한 landmark |
| --- | --- |
| 측면 구도 검사 보강 (2-3-5 5-4절) | hip, shoulder, ear, eye, nose 등 상체 정렬 분석 |
| 손-팔 스윙 간섭 분석 | wrist, elbow 등 상지 landmark |
| 신체 bbox 또는 scale 계산 | nose ~ foot_index 전체 |
| Overlay 시각화 디버깅 | 33개 전체 |

본 모듈은 33개 전체 landmark를 내부 버퍼에 일시 보관하며, **디버그 플래그가 활성화된 경우** `landmarks_full` 필드로 후속 모듈에 추가 전달한다. **MVP 운영 모드에서는 메모리 절약을 위해 `landmarks_full`을 채우지 않으며**, 6종(`landmarks`)만 전달한다.

### 4-2. 추출 실패 처리

`pose_detected = False`인 경우 (MediaPipe가 사람을 검출하지 못한 경우):

- `landmarks` 필드는 비어있는 객체 또는 누락
- `landmarks_full`도 (디버그 모드라도) 비어있음
- 후속 단계는 해당 frame을 분석에서 제외

> **추출 실패의 의미**: MediaPipe Pose가 사람을 검출하지 못한 frame은 `pose_detected = False`로 표시된다. 이 frame이 누적될 경우 2-3-5의 `target_lost` 또는 `low_landmark_visibility` reason code가 트리거된다.
>

## 5. 처리 비용

WebSocket 실시간 스트림 모드에서는 frame당 처리 시간이 30fps 한 frame 예산(33ms) 안에 들어와야 한다. 본 단계의 추정 처리 비용은 다음과 같다.

| 단계 | 추정 시간 (per frame) | 비고 |
| --- | --- | --- |
| MediaPipe Pose 추론 | ~33ms | `full` 모델 기준 (CPU) |
| Landmark 결과 파싱 + 6종 추출 | ~1ms | 33개 중 6종 선택 |
| **본 단계 합계** | **~34ms** |  |

본 추정치는 설계 단계의 예상치이며, 구현 단계에서 실측 검증한다. `lite` 모델 채택 또는 GPU 가속 시 시간이 절반 이하로 단축될 수 있다.

### 2-3-2와의 통합 latency

| 단계 | 추정 시간 (per frame) |
| --- | --- |
| 2-3-2 영상 전처리 + Frame-level 품질 검사 | ~15ms |
| 본 단계 (MediaPipe Pose 추출) | ~34ms |
| **합계** | **~49ms** |

단순 합산 49ms는 30fps 예산(33ms)을 **초과**한다. 따라서 **실시간 처리는 다음 중 하나를 전제로 한다**:

- **비동기 파이프라인**: 2-3-2와 본 단계를 별도 thread/process로 분리하여 병렬 처리
- **30fps → 15fps adaptive inference**: timestamp 기반 IC 계산을 유지하면서 추론 fps만 절반으로 낮춤
- **`lite` 모델 + GPU 가속**: 추론 시간 자체를 50% 이상 단축

통합 latency의 실측 검증은 데모 환경에서 진행하며, 그 결과에 따라 최종 운영 정책을 확정한다.

> **시간 예산 초과 시 대응 우선순위**: 1차로 비동기 파이프라인 채택을 시도하고, 그래도 latency가 부족하면 2차로 `lite` 모델 / GPU 가속을 검토한다. 3차 대안으로 adaptive inference를 사용하지만, 이 경우 IC 검출 시점 분해능이 절반으로 떨어지므로 마지막 수단으로 둔다.
> 

### Adaptive inference 향후 확장 영역

운영 환경의 처리 성능이 부족한 경우, 다음 adaptive 정책으로 확장 가능하다.

| 정책 | 설명 | 영향 |
| --- | --- | --- |
| 다중 품질 플래그 frame 추출 생략 | 2-3-2의 `low_brightness` + `motion_blur` + `frame_unstable` 동시 활성화 frame은 Pose 추출 생략 | 추출 성공률 약간 감소, 처리 비용 절약 |
| Backlog drop | 처리 backlog가 일정 이상 쌓이면 최신 frame만 유지하고 오래된 frame drop | 시간 정확도 유지, 분석 frame 수 감소 |
| 30fps → 15fps adaptive inference | 입력은 30fps 그대로 받되 추론은 15fps로 낮춤 | IC 검출 시점 분해능 절반 감소 (ms 단위 정확도 영향) |

MVP에서는 단순성 우선으로 모든 frame에 추출을 시도하며, 위 adaptive 정책은 **실측 latency가 운영 가능 범위를 벗어날 때 단계적으로 도입**한다.

## 7. 임계값 및 모델 선택 검증 계획

본 문서가 정의한 모델 선택 및 파라미터의 타당성을 실측 데이터로 검증하기 위한 실험. 이 실험은 데모 영상 수집 후 진행한다.

| 실험 항목 | 방법 | 검증 대상 |
| --- | --- | --- |
| `lite` vs `full` 모델 정확도 비교 | 동일 영상에서 두 모델의 heel / foot_index visibility 분포 비교 | 모델 선택 (3-2절) |
| `lite` vs `full` 처리 시간 측정 | 30fps frame당 평균 처리 시간 | 처리 비용 (5장) |
| `pose_detected = True` 비율 측정 | 정상 영상 / 다중 인물 영상 / 어려운 환경 영상에서 추출 성공률 | 모델 robustness 평가 |
| Visibility 분포 측정 | 정상 영상에서 6종 landmark의 visibility 평균/최솟값 | 2-3-5 임계값(0.6) 타당성 검증 자료 제공 |
| Live stream mode callback 순서 검증 | 의도적으로 frame 처리 시간을 변동시켜 callback 순서 측정 | 3-5절 *"순서 보장 안 됨"* 가정의 실제 영향 정량화 |
| 본 단계 + 2-3-2 통합 latency 측정 | end-to-end frame 처리 시간 측정 | 5장 비동기 파이프라인 / adaptive inference 필요성 판단 |

> 자세 지표(Foot Strike, Knee Flexion, Trunk Lean) 자체의 분류 임계값 검증 실험은 2-3-4 문서의 검증 계획을 따른다. 본 문서는 MediaPipe 사용 자체에만 집중한다.

## 8. 최종 정리

본 문서는 AI 파트의 MediaPipe Pose 사용 정책의 단일 정답 위치이다.

핵심 결정 사항을 다시 정리하면 다음과 같다.

1. **MediaPipe Pose Tasks API**의 Pose Landmarker를 사용한다.
2. **운영 환경은 Live stream mode**, 검증/실험 환경은 Video mode를 사용한다.
3. **`num_poses = 1`** 단일 인물 모드. 한 명의 pose만 반환되며, 이 pose가 분석 대상자라는 보장은 없다. 분석 대상자 전환은 2-3-5의 추적 안정성 정책으로 위임한다.
4. **모델은 `full`을 default로 두고, `lite`와 비교 평가 후 확정한다.** 평가 기준은 visibility 안정성 / 추출 성공률 / 처리 시간 3가지이며, visibility 안정성이 최우선이다.
5. **6종 landmark만** 후속 단계로 전달한다 (shoulder, hip, knee, ankle, heel, foot_index 각 좌우). 디버그 모드에서는 33개 전체를 `landmarks_full`로 추가 전달한다.
6. **Normalized 좌표 사용**, world landmark는 본 v1에서 사용 안 함.
7. **모든 frame에 대해 추출 시도**하되, 실패는 `pose_detected = False`로 표시.
8. **Timestamp는 2-3-2가 부여한 capture timestamp를 그대로 passthrough**한다. 본 모듈은 timestamp를 변경하지 않으며, 단일 정답 위치는 2-3-2 문서 3-3절이다.
9. 처리 비용은 frame당 약 34ms로 추정되며, 2-3-2와 통합 latency는 49ms로 30fps 예산을 초과한다. **실시간 처리는 비동기 파이프라인 또는 adaptive inference를 전제**로 한다.

Smoothing, 분석측 선택, visibility 임계 평가 등은 모두 다른 문서(2-3-4, 2-3-5)가 담당한다. 본 문서는 MediaPipe 호출과 결과 전달까지만 책임진다.
